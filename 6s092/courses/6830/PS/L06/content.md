# Indexing and Access Methods

Indexes speed up data access by maintaining auxiliary data structures over one or more attributes. The two dominant index types are **B+ trees** (for range and equality queries) and **hash indexes** (equality only). Understanding their structure, costs, and trade-offs is fundamental to query optimization.

## B+ Tree Indexes

A **B+ tree** is a balanced tree where all data pointers reside in leaves. Inner nodes hold $d$ to $2d$ keys (order $d$). Leaves are linked in a doubly linked list for efficient range scans. Insert and delete maintain balance by splitting or merging nodes.

<checkyourself>
Why are data records stored only in leaf nodes of a B+ tree (not inner nodes), unlike a B-tree?

<showhide>
Storing data only in leaves lets inner nodes be kept entirely in memory as a "find-the-leaf" index. Inner nodes can then hold more keys (since they store only keys, not full records), reducing tree height. The linked leaf list also enables efficient sequential scans without revisiting inner nodes.

## Hash Indexes

A **static hash index** partitions records into $N$ buckets using $h(k) \\bmod N$. **Extendible hashing** and **linear hashing** handle overflow by dynamic bucket splitting. Hash indexes support only equality lookups — range queries require a scan.

## Clustered vs. Unclustered Indexes

A **clustered** index keeps the data file sorted by the index key. Range scans on the key require reading a contiguous range of pages. An **unclustered** index may point to records scattered across many pages — in the worst case, one I/O per matching record.
