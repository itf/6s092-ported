# Buffer Pool Design and Memory Management

The **buffer pool** is the DBMS's in-memory cache of disk pages. It is the central component that all other DBMS modules use to read and write data. Effective buffer pool management reduces I/Os by keeping frequently accessed pages in memory.

## Buffer Pool Structure

The buffer pool consists of $B$ **frames**, each capable of holding one page. A **page table** maps page IDs to frames. Each frame has a **dirty bit** (page has been modified and must be written back before eviction) and a **pin count** (number of threads currently using the page — pinned pages cannot be evicted).

<checkyourself>
Why must a page with a non-zero pin count not be evicted from the buffer pool, even if its dirty bit is set?

<showhide>
A non-zero pin count means at least one thread holds a reference to that page and is actively reading or writing it. Evicting the page would invalidate those references and corrupt the thread's view of the data. The thread must first unpin the page (decrement pin count to 0) before the buffer manager is allowed to evict it.
</showhide>
</checkyourself>

## Replacement Policies

When all frames are pinned or the pool is full, a **replacement policy** selects a victim frame to evict (writing it to disk if dirty). Common policies:

- **LRU (Least Recently Used)**: evict the frame not used for the longest time.
- **Clock**: an approximation of LRU using a reference bit; sweep a "clock hand."
- **MRU (Most Recently Used)**: useful for sequential scans (the just-read page won't be needed again soon).

## Pre-fetching and Write-back

**Pre-fetching** (read-ahead) issues I/Os for pages predicted to be needed soon, hiding disk latency. For sequential scans this is straightforward. For index scans, the DBMS can pre-fetch leaf pages of a B+ tree. **Write-back** (background flushing) writes dirty pages to disk proactively, reducing the latency of future evictions.
