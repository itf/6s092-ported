# Database Operators and Query Processing

A query is translated into a **query plan** — a tree of relational operators. Each operator reads its input (from disk or from child operators), processes tuples, and produces an output stream. Understanding iterator (volcano) model and the I/O costs of basic operators is central to query optimization.

## The Iterator (Volcano) Model

Every operator implements three methods: `open()`, `get_next()`, and `close()`. `get_next()` returns one tuple at a time (or EOF). This **pipeline** model avoids materializing large intermediate results and enables operators to be composed without knowing the implementation of their children.

<checkyourself>
What is the advantage of the volcano/iterator model over materializing full intermediate results between operators?

<showhide>
Pipelining avoids writing intermediate results to disk. Tuples flow one-at-a-time (or in small batches) from child to parent, keeping memory usage low. The cost of materializing a large intermediate relation could dominate query cost — pipelining amortizes this. The trade-off is that some operators (sort, hash join) require blocking and must still buffer data.
</showhide>
</checkyourself>

## Selection and Projection

**Selection** $\\sigma_p$ filters tuples satisfying predicate $p$. Cost = one full scan = $N$ I/Os (no index). **Projection** $\\pi_L$ retains only listed attributes. Projection is free in a pipeline but requires an extra pass if duplicate elimination is needed (sort or hash).

## Sorting

External merge-sort sorts data larger than memory. With $B$ buffer pages and $N$ data pages: **Pass 0** creates $\\lceil N/B \\rceil$ sorted runs; each subsequent pass merges $B-1$ runs into one. Total passes = $1 + \\lceil \\log_{B-1} \\lceil N/B \\rceil \\rceil$. Total I/O = $2N \\times \\text{(number of passes)}$.
