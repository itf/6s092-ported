# Introduction to Database Internals

A DBMS must store data on disk and bring it into memory for processing. The fundamental unit of I/O is a **page** (typically 4–16 KB). Understanding storage layout, heap files, and the cost model for I/O is essential for analyzing query performance.

## Disk Storage and the Cost Model

Disk I/O is orders of magnitude slower than RAM access. The cost model counts **page I/Os** rather than CPU operations. A relation stored as a **heap file** consists of an unordered collection of pages, each containing a set of records.

<checkyourself>
If a disk has a seek time of 5 ms, rotational latency of 4 ms, and transfer rate of 100 MB/s for an 8 KB page, estimate the time to read one random page. How does this compare to sequential throughput?

<showhide>
Random read: seek (5 ms) + rotation (4 ms) + transfer (8 KB / 100 MB/s ≈ 0.08 ms) ≈ 9 ms per page. Sequential throughput: 100 MB/s = 12,500 pages/s, so ~0.08 ms/page. Random I/O is roughly 100× slower than sequential I/O.

## Heap Files and Page Layout

A **heap file** supports insert (append to last page), delete (mark slot as free), and scan (read all pages). Each page has a **header** (free space pointer, slot directory) and a **data area** storing fixed- or variable-length records.

## Record Layout

**Fixed-length records** are easy to store and access by offset. **Variable-length records** (e.g., VARCHAR fields) require a slot directory or offset array in the page header to locate each record, enabling efficient direct access and deletion without compaction.
