# Join Algorithms

Joins are among the most expensive operations in query processing. The three main join algorithms are **nested-loop join**, **sort-merge join**, and **hash join**. Their I/O costs differ dramatically, and the right choice depends on buffer size, data size, and whether inputs are sorted or indexed.

## Nested-Loop Join

**Simple nested-loop join** (SNLJ): for each tuple in $R$ (outer), scan all of $S$ (inner). Cost = $|R| + |R_{tuples}| \\cdot |S|$ page I/Os — prohibitive for large relations.

**Block nested-loop join** (BNLJ): read $B-2$ pages of $R$ at a time, scan all of $S$ once per chunk. Cost = $|R| + \\lceil |R|/(B-2) \\rceil \\cdot |S|$.

<checkyourself>
In BNLJ, why should the **smaller** relation be the outer (left) relation?

<showhide>
The inner relation $S$ is scanned once per chunk of the outer relation. If the outer has $|R|$ pages and chunk size is $B-2$, $S$ is scanned $\\lceil |R|/(B-2) \\rceil$ times. Making $R$ the smaller relation minimizes this count, reducing total I/Os.
</showhide>
</checkyourself>

## Sort-Merge Join

Sort both relations on the join key, then merge by scanning both in order. **Cost** = sort $R$ + sort $S$ + merge pass. Merge cost = $|R| + |S|$. If both fits in $\\sqrt{B}$ pages or relations are pre-sorted, this can be very efficient. Sort-merge also produces output sorted on the join key.

## Hash Join

**Grace hash join**: partition both $R$ and $S$ into $k$ partitions using the same hash function $h_1$ on the join key. Then, for each partition $i$, load partition $R_i$ into memory (build phase) and probe with $S_i$ tuples. Cost = $3(|R|+|S|)$ if each partition of $R$ fits in $B-2$ memory pages (requires $B \\geq \\sqrt{|R|}$).
