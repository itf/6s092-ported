# Introduction to Database Systems

Database systems manage large collections of persistent, structured data and provide efficient query, update, and transaction capabilities. This lecture surveys the landscape of database systems: why we need them, what they provide beyond file systems, and the key abstractions they introduce.

## Why Databases?

File systems lack critical features that databases provide: declarative querying (describe *what* you want, not *how* to get it), data independence (change storage without changing applications), concurrent multi-user access with isolation, and crash recovery.

<checkyourself>
What is the difference between **physical data independence** and **logical data independence** in a DBMS?

<showhide>
Physical data independence means applications are insulated from changes in how data is stored on disk (e.g., switching from a heap file to a B-tree index). Logical data independence means applications are insulated from changes in the logical schema (e.g., splitting one table into two). Logical independence is harder to achieve because views that expose the old schema must faithfully support updates.

## The ANSI/SPARC Three-Level Architecture

DBMSs are structured in three levels: the **external** (view) level seen by users, the **conceptual** (logical) level describing all data, and the **internal** (physical) level describing actual storage. Mappings between levels provide the two forms of data independence.

## Data Models

A **data model** is a collection of concepts for describing data. The dominant model today is the **relational model** (tables of rows/columns). Earlier models include hierarchical and network; newer ones include document, graph, and key-value stores.
