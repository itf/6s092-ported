# The Relational Model

The relational model, introduced by E.F. Codd in 1970, represents data as sets of tuples organized into relations (tables). It provides a clean mathematical foundation and a powerful declarative query language. Understanding the formal model — relations, schemas, keys, integrity constraints, and relational algebra — is prerequisite to everything else in database systems.

## Relations, Schemas, and Instances

A **relation schema** $R(A_1{:}D_1, \ldots, A_n{:}D_n)$ names a relation and its attributes with their domains. A **relation instance** is a finite set of tuples, each mapping attribute names to domain values. The **degree** (arity) is $n$; the **cardinality** is the number of tuples.

<checkyourself>
Why does the relational model treat a relation as a **set** (no duplicates, no order) rather than a list? What practical consequences does this have for SQL?

<showhide>
Using a set gives clean set-theoretic semantics and avoids representation redundancy. In practice SQL uses *multisets* (bags) for efficiency — eliminating duplicates requires sorting or hashing. The consequence is that SQL SELECT without DISTINCT can return duplicate rows, and query results have no guaranteed order without ORDER BY.

## Keys and Integrity Constraints

A **superkey** is a set of attributes whose values uniquely identify every tuple. A **candidate key** is a minimal superkey. The chosen candidate key is the **primary key**. A **foreign key** in relation $R$ referencing relation $S$ requires every value in the FK attributes to appear as a primary key value in $S$ (referential integrity).

## Relational Algebra

Relational algebra provides six core operators: **selection** $\\sigma$, **projection** $\\pi$, **Cartesian product** $\\times$, **union** $\\cup$, **set difference** $-$, and **rename** $\\rho$. Additional derived operators include natural join $\\bowtie$, theta-join, and division.
