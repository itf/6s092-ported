# I/O Concurrency

A high-performance server must overlap I/O waits with useful computation. When a server waits for a disk read or a network transfer, its CPU is idle—wasted capacity. **I/O concurrency** is the ability to overlap these waits with useful work on behalf of other clients.

The lecture surveys three software structures for achieving I/O concurrency: (1) multiple processes (one per client), (2) multiple threads within a single process, and (3) event-driven (non-blocking I/O with a select/poll loop). Each approach has different trade-offs in terms of complexity, context-switch overhead, and interaction with OS facilities.

For a web server, the performance benefits can be enormous. If a disk read takes 10 ms and cached responses take 0.1 ms, a concurrent server can serve ~100 other clients during a single disk wait. Without concurrency, throughput is limited to 100 requests/second; with concurrency it can approach much higher rates.

## Multiple Processes and Threads

The simplest approach is to fork a new process per client. The OS provides I/O concurrency transparently: while one process blocks on a disk read, the scheduler runs another. The cost is high: fork is expensive, each process has its own address space (no shared memory), and thousands of processes create scheduling and memory pressure.

Threads are lighter-weight: they share an address space but have separate stacks and CPU state. Thread creation is faster than fork, and threads can share data structures directly. However, shared memory introduces races and requires locking.

<showhide>

Multi-thread: shared address space allows easy data sharing and lower creation cost, but requires careful locking to avoid data races. Threads are generally preferred for server applications that share a large working set.

## Event-Driven Programming

The event-driven approach uses a single thread with non-blocking I/O. The server registers interest in events (socket readable, writable, timer expired) with `select()` or `poll()`. When an event fires, the server runs the corresponding handler to completion before handling the next event. No context switches are needed, and there is no stack-per-client overhead.

The price is programming complexity: the programmer must manually implement what a thread's stack provides automatically—saving and restoring per-client state across handler calls. This is the **callback hell** or state-machine style of programming.

Why does an event-driven server not need locks for its own data structures, even when handling many concurrent clients?

<showhide>
Because the event loop runs handlers one at a time to completion—there is no preemptive switching between handlers. At any moment, only one handler is executing, so no two handlers access shared data simultaneously. This eliminates the need for locks on server-side data structures (though you still need locks if you introduce helper threads for blocking operations like disk I/O).
