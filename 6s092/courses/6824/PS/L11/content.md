# Vector Timestamps and Version Vectors

Vector timestamps (also called vector clocks) solve the problem of tracking causality in distributed systems: given two events at different nodes, can we determine which happened first, or are they concurrent? **Version vectors** apply the same idea to file synchronization: given two replicas of a file, which is newer, or have they diverged (concurrent modifications requiring human resolution)?

These ideas arise wherever "optimistic replication" is used—replicas operate independently (perhaps disconnected) and synchronize periodically. The challenge is distinguishing "A updated after B" (take A's version) from "A and B updated independently" (conflict requiring merge or human intervention).

The lecture uses the example of a file synchronizer for laptops: each laptop has a full copy of a set of files, modifications happen offline, and synchronization resolves differences.

## Vector Clocks for Causality

A **vector clock** at node $i$ is a vector $V_i$ of $n$ integers (one per node). Node $i$ increments $V_i[i]$ on each local event. When node $i$ sends a message, it includes its current vector. When node $j$ receives a message with vector $V$, it updates $V_j[k] = \max(V_j[k], V[k])$ for all $k$, then increments $V_j[j]$.

Event $a$ **happened before** event $b$ ($a \to b$) if and only if $V_a[k] \leq V_b[k]$ for all $k$ and $V_a \neq V_b$. If neither $V_a \leq V_b$ nor $V_b \leq V_a$ (they are incomparable), the events are **concurrent**.

## Version Vectors for File Synchronization

A **version vector** for a file is a vector of (node, timestamp) pairs recording the last write seen from each node. File $f$ at node A has version vector $VV_A(f)$. When A and B synchronize:
- If $VV_A(f) \leq VV_B(f)$: B's version is strictly newer; copy B→A.
- If $VV_B(f) \leq VV_A(f)$: A's version is strictly newer; copy A→B.
- Otherwise: concurrent modification; report conflict.

The version vector approach correctly detects all conflicts and does not produce false conflicts (spurious conflict reports for non-conflicting changes).

What is the difference between a vector clock and a version vector, and in what contexts is each used?

<showhide>
A **vector clock** is attached to individual events (messages, operations) and tracks causal dependencies between events across nodes—it answers "did event A happen before event B?" **Version vectors** are attached to data objects (files, database records) and track the history of writes to that object—they answer "is version A of this file a descendant of version B, or did they diverge?" Version vectors are essentially vector clocks applied to object versions rather than arbitrary events. TreadMarks uses vector timestamps to track which write notices each processor has received (similar to version vectors for pages).
</showhide>
