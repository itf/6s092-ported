# Harp: Replication in the Harp File System

Harp (Liskov et al., SOSP 1991) is a complete, production fault-tolerant NFS server built using Viewstamped Replication. It demonstrates how VSR's theory can be turned into a practical system with high performance. Key innovations include: a **write-ahead log** for both in-memory state and on-disk recovery, **witnesses** that reduce storage requirements, and support for concurrent overlapping operations.

Harp uses $2b+1$ servers (primary, $b$ backups, and $b$ witnesses). Witnesses hold no file data normally—they only store log entries during view changes, allowing the system to tolerate $b$ failures while keeping storage costs low. The system can survive a simultaneous power failure of all $2b+1$ servers by replaying the log from disk.

## Harp Architecture and Log Design

The heart of Harp is its **in-memory log**. The log tracks operations in flight and supports several pointers: FP (front pointer, most recent client request), CP (commit point, operations that have been committed), AP (disk write pointer), LB (disk completion pointer), GLB (global lower bound—all nodes have completed disk writes up to here).

The gap between FP and CP allows multiple concurrent operations to be in flight simultaneously: the primary can pipeline requests to backups without waiting for each to complete before sending the next. This increases throughput significantly compared to a one-at-a-time protocol.

<showhide>
A witness participates in view changes and stores log entries—but under normal operation it does not store any file data. This reduces storage costs: instead of $2b+1$ full copies of the file system, you only need $b+1$ full copies (primary + backups). Witnesses only need to hold the log, which is much smaller. In a view change, the witness provides its log entries so the new primary can reconstruct any operations that were in flight when the old primary crashed.

## Recovery and the GLB

After a failure (including simultaneous power failure of all servers), Harp recovers by replaying the log. The **GLB** (global lower bound) marks the point up to which all servers have written to disk. On recovery, each server replays log entries from GLB to CP, bringing its disk state up to date. This is faster than copying the entire file system.

The log's role during recovery: it bridges the gap between the on-disk state (accurate up to GLB) and the last committed in-memory state (at CP). Without the log, a server that crashes mid-operation would have inconsistent disk state.

What is the FP-CP gap in Harp's log, and how does it improve performance over a simple one-at-a-time replication protocol?

<showhide>
FP (front pointer) marks the most recently received client request; CP (commit point) marks the most recently committed operation. The gap FP−CP represents operations that have been sent to backups but not yet ACK'd and committed. By allowing multiple operations to be in flight simultaneously (pipelining), the primary can keep backups' networks and disks busy, increasing throughput. In a one-at-a-time protocol, the primary would wait for each operation to be committed before sending the next, wasting the round-trip latency between each operation.
