# RPC Transparency

The goal of **RPC (Remote Procedure Call)** is transparency: a programmer writing a distributed system should be able to call remote procedures as if they were local, with no change to the calling code. NFS is used as a case study to evaluate how well this ideal is achieved in practice.

True RPC transparency turns out to be an illusion. While the API (function signatures) can be preserved, the **semantics** of remote calls differ fundamentally from local calls: latency is orders of magnitude higher, calls can fail in new ways (network loss, server crash, partial completion), and retrying a failed call may execute the operation twice. These differences force application programmers to think about distribution even when using an RPC library.

The lecture examines RPC failures, retry semantics, idempotency, and the cost of maintaining illusions of locality in systems like NFS.

## Failure Modes and Retry Semantics

A local function call either returns a result or raises an exception—it never silently fails. An RPC can fail in many more ways: the request may be lost, the reply may be lost (but the operation completed), the server may crash mid-execution, or the network may be partitioned. The client cannot distinguish "server crashed before processing" from "reply lost after processing."

The client must choose a retry policy. **At-least-once** semantics retry until a reply arrives—safe only for idempotent operations. **At-most-once** semantics use the server to detect and suppress duplicates. **Exactly-once** is the ideal but is impossible in the presence of server crashes without additional coordination (see Two-Phase Commit).

<checkyourself>
Why is it impossible to achieve exactly-once RPC semantics in the presence of server crashes, without additional persistent coordination (e.g., a log)?

<showhide>
If the server crashes after executing the operation but before persisting a record that it did so, and then restarts, it cannot tell whether it already ran the operation. It will re-execute on the client's retry, giving at-least-once semantics. Achieving exactly-once requires the server to durably record (before replying) that it handled the request, so that after a restart it can suppress duplicates. This requires a log or stable storage—at which point you have the beginnings of crash recovery (see Lecture 6).

## NFS as a Case Study in Transparency

NFS preserves the client-side API (POSIX system calls) but cannot fully preserve semantics. Three key deviations: (1) NFS requires user-IDs to be passed explicitly in each RPC (no implicit process identity), (2) NFS file handles replace file descriptors (handles survive server reboots, descriptors do not), and (3) NFS requires synchronous writes for durability (local file systems can batch writes).

The server-side implementation also required changes: in-kernel threads to serve concurrent RPCs, generation numbers added to i-nodes on disk, and flag arguments to force synchronous disk updates.

<checkyourself>
What does it mean for an RPC operation to be idempotent, and why does NFS require its operations to be idempotent?

<showhide>
An operation is idempotent if executing it multiple times has the same effect as executing it once. NFS requires idempotency because it uses at-least-once retry: if a reply is lost, the client resends the request. If the operation is idempotent, the retry is harmless. READ, WRITE (to a fixed offset), LOOKUP, and GETATTR are all idempotent. Operations like CREATE and REMOVE require special handling (e.g., ignore EEXIST on retried CREATE, ignore ENOENT on retried REMOVE).
