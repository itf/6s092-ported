# Scalable Lookup: Distributed Hash Tables and Chord

The previous lectures focused on "machine-room" distributed systems with a small, known, centrally managed set of servers. This lecture turns to **Internet-scale** systems where nodes are owned by arbitrary individuals, the population is huge (millions), and no central authority maintains a complete membership list.

The key challenge: **scalable lookup**. Given a key (e.g., a file name, a user ID, a piece of content), how do you efficiently find the node that stores the corresponding value? The naive approach—contact a central server—works (Napster used it) but creates a bottleneck and single point of failure. **Distributed Hash Tables (DHTs)** solve this with $O(\log n)$ hops and no central node.

The lecture focuses on **Chord** (Stoica et al., 2001), which organizes nodes in a consistent hash ring and provides $O(\log n)$-hop routing to any key using **finger tables**.

## Consistent Hashing and the Chord Ring

In Chord, both nodes and keys are assigned IDs in a circular namespace $[0, 2^m)$ using a hash function (e.g., SHA-1). Each key is stored at its **successor**: the first node whose ID is $\geq$ the key's ID in the ring. When a node joins, it takes responsibility for a subset of keys from its successor.

**Consistent hashing** minimizes disruption on node joins/leaves: on average, only $K/n$ keys move when a node joins a system with $n$ nodes and $K$ total keys ($K/n$ is the average load per node).

Similarly, when a node leaves, its keys move only to its successor. With $n$ nodes and $K$ keys, only $K/n$ keys move per join/leave event, compared to $O(K)$ moves for naive hash tables.

## Finger Tables and O(log n) Routing

Without finger tables, routing in Chord requires traversing the ring one hop at a time—$O(n)$ hops in the worst case. **Finger tables** add shortcuts: node $i$'s finger table entry $k$ points to the successor of $i + 2^k \pmod{2^m}$. With $m$ bits, each node has $m = O(\log n)$ fingers.

Lookup: to find the successor of key $k$, forward the query to the largest finger whose ID is $< k$. Each hop covers at least half the remaining distance, so the query reaches the target in $O(\log n)$ hops.

What are the main limitations of DHTs like Chord for building real applications, and how do systems like BitTorrent work around them?

<showhide>
DHTs provide only a `put(key, value)` / `get(key)` interface with loose durability guarantees. Limitations: (1) no complex queries (range queries, full-text search), (2) data availability depends on nodes staying online—if the responsible node leaves and no replication is done, data is lost, (3) security/trust: any node can store incorrect values for a key. BitTorrent works around these by using DHTs (Kademlia) only for peer discovery (finding peers that have a torrent), while actual file data is replicated across many peers with content hashing for verification. The DHT stores (infohash → list of peers), not the file itself.
</showhide>
