# Event-Driven Programming: libasync and RPC

This lecture develops event-driven programming further, introducing the `libasync` and `libarpc` libraries used in the course labs. Rather than writing raw `select()` loops, programmers use a higher-level interface: `libasync` buffers I/O and translates low-level socket events into application-level callbacks; `libarpc` extends this to RPC calls, so that sending an RPC and receiving its reply are expressed as an asynchronous call with a callback.

The lab architecture is a distributed file system (`ccfs`) built on top of a block server and a lock server. Writing RPC client code, async functions that call RPCs, and RPC server handlers all follow the same callback-continuation idiom: each potentially-blocking step is split into "send request" and "handle reply" phases.

The Flash web server (Pai et al., 1999) is presented as a case study combining event-driven I/O with helper processes for blocking disk operations, achieving high performance comparable to in-kernel web servers.

## The libasync Programming Model

In `libasync`, I/O never blocks the event loop. A write to a socket appends data to an output buffer; the library drains the buffer asynchronously. An RPC call via `libarpc` sends the request and registers a callback; the event loop invokes the callback when the reply arrives. The programmer writes code as a chain of callbacks rather than a sequential function.

This structure forces explicit management of per-request state—state that threads would store on the stack is instead stored in heap-allocated closures or objects passed between callbacks.

<showhide>
Because each callback returns to the event loop immediately after registering the next step. The original stack frame is gone by the time the next callback fires. Per-request state must therefore live in the heap (e.g., a closure, a struct, or a `ref`-counted object) so it persists across the gap between callbacks.

## Flash: Asymmetric Multi-Process Event-Driven Server

Flash combines an event-driven core with blocking helper processes. The main process uses `select()` for all network I/O. For disk reads (which cannot be done non-blocking on most UNIX systems), it dispatches to a pool of helper processes. Helpers block on disk, then signal the main process via a pipe. This preserves the single-threaded simplicity of the event loop for most work while still achieving disk I/O concurrency.

Flash also uses application-level caching and pathname caching to avoid repeated disk lookups, and it memory-maps files for zero-copy transfers to the network. The result is performance competitive with in-kernel servers (e.g., khttpd) while remaining in user space.

What is the advantage of event-driven servers over thread-per-request servers for handling a large number of mostly-idle connections (e.g., long-polling HTTP clients)?

<showhide>
Event-driven servers have O(1) memory cost per idle connection (just a file descriptor in the select/epoll set), whereas thread-per-request servers allocate a full thread stack (typically 1–8 MB) per connection. With 10,000 idle connections, threads would consume 10–80 GB of memory; an event-driven server uses negligible memory for idle connections.
