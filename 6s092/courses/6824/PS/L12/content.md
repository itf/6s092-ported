# Two-Phase Commit

Two-phase commit (2PC) is the standard protocol for **atomic distributed transactions**: either all participating servers commit an operation, or none do. The motivating example is a bank transfer—debit at bank A, credit at bank B—where we must never have one succeed without the other.

2PC introduces a **Transaction Coordinator (TC)** that orchestrates the protocol. In the first phase, the TC asks all participants whether they can commit (**prepare**). In the second phase, if all replied YES, the TC sends **commit**; otherwise it sends **abort**. Participants that vote YES must be prepared to commit or abort based on the TC's final decision, even if they crash and restart.

The key property: 2PC guarantees atomicity even in the presence of crashes, but it is **blocking**—if the TC crashes after participants have voted YES but before it sends the decision, participants must wait for the TC to recover. They cannot unilaterally decide.

## The Two-Phase Commit Protocol

**Phase 1 (Prepare):** TC sends PREPARE to all participants. Each participant durably logs "prepared" and replies YES or NO. Voting YES is a promise: the participant can no longer unilaterally abort.

**Phase 2 (Commit/Abort):** If all voted YES, TC logs COMMIT and sends COMMIT to all. Otherwise, TC logs ABORT and sends ABORT to all. Participants log their final decision and acknowledge.

Recovery: the TC's log determines the outcome. If the TC logged COMMIT before crashing, it re-sends COMMIT on restart. If it logged PREPARE but not COMMIT (or ABORT), it can decide: if all YES then commit, else abort.

## Blocking and Fault Tolerance

The critical weakness of 2PC: if the TC crashes after participants vote YES, participants are **blocked**—they hold locks and cannot release them until the TC recovers. This limits availability: the system is unavailable for transactions involving the blocked resource until the TC restarts.

Three-phase commit (3PC) attempts to address this with an additional phase that allows participants to recover without the TC, but it requires assumptions (no network partitions) that are unrealistic in practice. In real systems, 2PC blocking is addressed by ensuring TC recovery is fast (persistent TC state on replicated storage).

How does the 2PC coordinator's log enable recovery after a crash, and what must be logged and when?

<showhide>
Before Phase 1: TC logs the transaction ID and participants. Before sending Phase 2 decision: TC logs COMMIT or ABORT. This is the commit point—once logged, the decision is final. On recovery, TC reads its log: if COMMIT is logged, resend COMMIT to all participants; if ABORT is logged, resend ABORT; if only the participant list is logged (no decision), TC can decide based on re-collecting votes (or abort if it can't reach all). Participants also log their vote (YES/NO) before replying and log COMMIT/ABORT when they receive the Phase 2 message.
