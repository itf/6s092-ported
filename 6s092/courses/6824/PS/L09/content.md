# Memory Consistency (Part 1)

Replicated data is a central theme in distributed systems. Replication is used for both performance (caches reduce latency) and fault tolerance (copies survive individual node failures). But replication introduces a fundamental question: what does a reader see when multiple writers exist? The answer is a **memory consistency model**.

This lecture develops consistency models in the context of shared-memory multiprocessors (loads and stores), where the hardware and compiler are free to reorder operations for performance. The same concepts apply directly to distributed systems: NFS caches, distributed shared memory, and file replication all face the same question of what reads are allowed to return.

The simplest model—**sequential consistency** (SC)—requires that all operations appear to execute in some global sequential order consistent with each processor's program order. SC is intuitive but hard to implement efficiently.

## Sequential Consistency

**Definition**: An execution is sequentially consistent if there exists some total ordering of all operations such that (1) all processors see results consistent with that total order (reads return the most recent write in the order), and (2) each processor's operations appear in their program order within the total order.

SC is the "natural" model: it matches what a programmer expects from a shared blackboard. The system is free to interleave instructions from different processors in any order, but once a value is revealed to a read, the system has committed to that partial order and must respect it transitively.

## Implementing Sequential Consistency

SC is expensive to implement because it forbids many hardware and compiler optimizations: write buffers, out-of-order execution, and compiler instruction reordering all violate SC in general. A processor with a write buffer may locally reorder a write to X followed by a read of Y, which can violate SC.

One implementation: treat each instruction as atomic and globally ordered. Every write is broadcast to all caches before the next instruction proceeds. This is a **sequentially consistent cache coherence protocol**—but it requires invalidation or update of all caches on every write, which is expensive on a bus and even more expensive in a distributed system.

How does sequential consistency relate to the consistency model provided by the NFS distributed file system?

<showhide>
NFS provides much weaker consistency than SC. SC requires that reads see the most recent write in some global order. NFS's close-to-open consistency only guarantees that a client sees another client's writes after the writer closes the file and the reader re-opens it. Between opens, clients may read arbitrarily stale cached data. This is analogous to a memory model that allows reads to return arbitrarily old values—far weaker than SC.
