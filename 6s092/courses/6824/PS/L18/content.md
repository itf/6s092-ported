# Scalable Internet Systems: Security, Availability, and Peer-to-Peer

This final lecture broadens the picture to large-scale Internet systems: peer-to-peer file sharing, content-addressable storage, and the security and availability challenges that arise when nodes are owned by untrusted, autonomous parties. In contrast to the machine-room systems of earlier lectures, Internet-scale systems must operate with no central authority, potentially malicious participants, and massive churn (nodes joining and leaving constantly).

The lecture examines how content-addressable systems (using cryptographic hashes as keys) provide integrity guarantees, and how reputation and redundancy substitute for trust in systems like BitTorrent and Freenet.

## Content-Addressable Storage

In a **content-addressable storage (CAS)** system, the key for a piece of data is the cryptographic hash of the data itself (e.g., SHA-1). A client that retrieves data for a key can verify its integrity by hashing the retrieved data and comparing to the key—any tampered or corrupted data will have a different hash.

This is the basis for the **Merkle tree** structure used in Git, BitTorrent, and many other systems: large files are split into blocks, each block is hashed, and the hashes are organized into a tree. The root hash authenticates the entire file; downloading blocks out of order is safe because each block's hash can be verified independently.

Why does using the cryptographic hash of content as its address provide integrity guarantees that a location-based address (e.g., an IP address and path) cannot?

<showhide>
A location-based address names where the data lives, not what the data is. A malicious or compromised server can serve incorrect data at that address with no way for the client to detect the substitution. A content hash names what the data is: the client verifies the retrieved data by computing its hash and comparing to the expected key. Any modification to the data changes its hash, so tampering is immediately detectable. The content hash is a tamper-evident address.

## Availability and Replication in Peer-to-Peer Systems

In a P2P system, individual nodes are unreliable—they go offline frequently and without warning (churn). To keep data available, it must be replicated across many nodes. A common heuristic: replicate to $r$ nodes; as long as at least one replica is online, the data is available. The probability of availability is $1 - (1-p)^r$ where $p$ is the probability a single node is online.

Maintaining replication in the face of churn requires a **repair protocol**: when a node detects that one of the replicas it is responsible for has gone offline, it creates a new replica on another node. Without repair, replicas gradually disappear as nodes leave.

What is the "sybil attack" in peer-to-peer systems, and why does it undermine DHT-based routing and storage?

<showhide>
A sybil attack is when a malicious party creates many fake node identities (sybils) in the P2P network. Since DHT routing is determined by node IDs, a sybil attacker can arrange its fake nodes to control the routing for any key—intercepting lookups, returning incorrect peer lists, or simply dropping requests. CAS mitigates sybil attacks for stored data (wrong data is detectable), but not for peer discovery (attacker can return a list of attacker-controlled peers). Defenses include proof-of-work for node ID generation, trusted certificate authorities for node IDs, or social-network-based trust.
</showhide>
