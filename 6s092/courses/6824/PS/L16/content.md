# Frangipani: A Scalable Distributed File System

Frangipani (Thekkath, Mann, Lee, SOSP 1997) is a distributed file system that achieves scalability by having each client machine run its own file server locally, with shared storage provided by the **Petal** distributed disk. Unlike NFS (one server for many clients), Frangipani has no file server bottleneck: clients access shared storage directly, coordinating through a distributed **lock service**.

The design exploits two key insights: (1) most operations can be performed locally if the client holds the appropriate locks, and (2) write-ahead logging per client provides crash recovery without a central coordinator. Frangipani is one of the earliest systems to combine distributed locking, per-client write-ahead logging, and a shared disk for scalable, consistent distributed file access.

## Architecture: Petal, Locks, and Per-Client Logs

Petal provides a virtual disk shared by all Frangipani servers (one per client machine). Each Frangipani server caches metadata locally, acquires locks on metadata objects before modifying them, and maintains a **write-ahead log in Petal** before writing the actual metadata. The log is stored in a dedicated region of Petal allocated to each server.

The lock service enforces **cache consistency**: when server S1 holds a lock on a directory and server S2 requests it, the lock service sends a REVOKE to S1. S1 must write its log and dirty metadata to Petal before releasing the lock, ensuring S2 sees up-to-date data.

<checkyourself>
In Frangipani, why must a server write its log to Petal before releasing a lock (rather than writing directly to the metadata region)?

<showhide>
Write-ahead logging requires the log record to be durable before the corresponding metadata update. If the server wrote directly to the metadata region and crashed mid-update, the metadata would be inconsistent. By writing the log first, recovery can either replay the log to complete the update or roll it back. If the server crashes after writing the log but before the metadata update, recovery finds the log and replays it. If it crashes before writing the log, the metadata is unchanged (consistent pre-update state).
</showhide>

## Consistency and Transactions via Locking

Frangipani provides multi-object atomic updates (e.g., creating a file requires updating both a directory block and an inode) by holding locks on all affected objects simultaneously. This is effectively a form of **two-phase locking**: acquire all locks before any writes, release after all writes. The WAL ensures atomicity with respect to crashes.

The combination of distributed locks + per-client WAL achieves strong consistency (linearizability of file operations) and crash recovery without a central file server—a rare combination.

How does Frangipani's recovery differ from traditional single-server file system recovery (e.g., ext3 with journaling)?

<showhide>
In ext3, the log and metadata are on the same local disk, and recovery is done by the local kernel after mounting. In Frangipani, the log is on shared Petal storage, and recovery is performed by a *different* Frangipani server acting as a recovery agent. This is necessary because the crashed server cannot recover itself. The recovery agent must be careful to replay only committed log entries (those with a complete commit record) and not partial ones. The fundamental structure (WAL, replay on recovery) is the same, but the distributed setting adds the complication of who performs recovery and how to safely release the crashed server's locks.
</showhide>
