# Cache Consistency and Locking

This lecture is a tutorial on cache consistency and distributed locking, in the context of the course's lab file system (`ccfs`). The goal is to understand why caching requires explicit coordination and how locks provide both correctness (mutual exclusion) and consistency (cache coherence).

The lab architecture has multiple `ccfs` server processes sharing a single block server. Without coordination, two servers can execute concurrent operations on the same directory—both fetch the block, add different entries, and write back—and one update is silently lost. The fix is to serialize conflicting operations using a **lock server**: a client acquires a lock before reading, holds it through the write, and releases it afterward.

The lecture also previews Lab 5, which adds caching: a client caches blocks locally and only contacts the block server when its cache misses. Correctness requires that a client invalidate its cached copy when another client acquires the lock on that block.

## Correctness: Atomicity via Locking

The standard definition of correctness for concurrent operations is **atomicity**: the result of any concurrent execution must be equivalent to some serial execution of the same operations. This is also called **serializability**. If all operations are serializable, you can reason about correctness as if operations happen one at a time.

The lock protocol: acquire lock → read block from server → modify → write block to server → release lock. This serializes all accesses to a block, making the read-modify-write atomic with respect to other clients.

## Performance: Caching with Lock-Based Invalidation

Without caching, every block read/write goes to the block server—the server becomes a bottleneck. With caching, a client keeps recently-read blocks locally and serves subsequent reads from cache. The catch: when another client modifies the block (and holds the lock), the cache must be **invalidated** before the lock is released, so the next reader fetches a fresh copy.

The combination of locks and cache invalidation provides both correctness and performance. If clients mostly access disjoint blocks, lock contention is rare and clients can serve most reads from their local cache. If clients frequently share blocks, lock revocations are frequent and the caching benefit diminishes.

What is the difference between a write-back cache and a write-through cache in the context of distributed locking, and which requires more careful coordination?

<showhide>
A **write-through** cache writes modifications immediately to the block server; the cached copy and the server copy are always in sync. When the lock is released, no flush is needed. A **write-back** cache defers writes to the block server until the lock is revoked or released. This reduces server traffic (multiple local modifications become one write) but requires the client to flush dirty data before releasing the lock. Write-back requires more coordination: the REVOKE handler must flush before acknowledging.
