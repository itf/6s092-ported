# Logging

Logging (write-ahead logging, WAL) is the standard solution to crash recovery for complex data structures. Instead of writing modified blocks directly to their on-disk locations, the system first appends a **log record** describing the intended update to a sequential log on disk. The log record is the **commit point**: once it is durably written, the operation is considered committed even if the machine crashes before the actual data blocks are updated. Recovery replays the log to bring the data blocks up to date.

Logging trades random writes (updating data blocks in place) for sequential writes (appending to the log), which is much faster on spinning disks. Recovery is fast too: instead of scanning the entire disk like `fsck`, recovery only needs to replay the log from the last checkpoint.

The lecture covers the write-ahead logging protocol, undo vs. redo logging, checkpointing, and the design of the ext3/LFS journaling file systems.

## Write-Ahead Logging Protocol

The WAL protocol has two rules: (1) before writing a dirty data block to disk, the corresponding log record must be on stable storage (**redo rule**); (2) before a transaction commits, all its log records must be on stable storage (**commit rule**). Together, these ensure recovery can always reconstruct a consistent state.

After recovery, the system replays all committed transactions from the log (redo) and rolls back all uncommitted transactions (undo). Log records must include enough information for both: redo records contain the new values; undo records contain the old values.

<showhide>

(2) **Commit rule**: all log records for a transaction must reach stable storage before the transaction is declared committed—so a committed transaction can always be fully replayed during recovery. Without the redo rule, we might lose an update that the user believes was committed. Without the commit rule, we might replay a partial transaction as if it were complete.

## Checkpointing and Log Truncation

Without truncation, the log grows forever. A **checkpoint** captures the current state: all dirty blocks are flushed to their data locations, a checkpoint record is written to the log, and log entries before the checkpoint are no longer needed for recovery. Recovery only needs to replay from the last checkpoint.

There is a trade-off: checkpointing requires flushing dirty buffers, which takes time and causes a burst of random writes. Systems often use **fuzzy checkpoints** that allow new transactions to proceed during the flush, at the cost of a more complex recovery procedure.

What is the difference between redo logging and undo logging, and when would you use each?

<showhide>
**Redo logging** records the new (after) value of each modified block. Recovery replays committed transactions to bring data blocks up to date. You use redo logging when you want to defer writing data blocks to disk (write the log first, write data later—good for batching). **Undo logging** records the old (before) value. Recovery rolls back uncommitted transactions. You use undo logging when you want to write data blocks before committing (write data, then write commit record). Most production systems use **undo-redo logging** (recording both old and new values) for maximum flexibility.
</showhide>
