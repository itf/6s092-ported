# Network File System (NFS)

The Network File System (NFS), developed at Sun Microsystems in the mid-1980s, is the canonical example of a distributed file system and a case study in the trade-offs of remote procedure call transparency. NFS allows client machines to mount a server's file system over a LAN and access files using standard UNIX system calls—`open`, `read`, `write`, `close`—with no application changes.

NFS's design choices have been enormously influential: the use of **stateless servers**, **idempotent operations**, and **file handles** are all direct responses to the challenge of crash recovery in a distributed setting. NFS also introduced important problems that later systems work to fix: weak consistency, performance loss from synchronous writes, and the awkwardness of stateless servers for operations like file locking.

The architecture is a client with a kernel-level NFS stub that translates VFS calls into RPC calls to the NFS server. The server executes the operations on its local disk and returns results.

## File Handles and Stateless Design

NFS servers are **stateless**: they do not remember which files are open or which clients are connected. Instead, every RPC carries a **file handle** encoding the file system ID, i-number, and a generation number. This means a server crash and restart is transparent—clients simply retry their RPCs until the server comes back.

The generation number prevents a stale handle from referring to a newly-created file that reused an old i-number. Without it, a client holding a handle to a deleted file might inadvertently access a new file that reused the same i-number on disk.

Why does NFS use file handles rather than embedding the file name in the RPC? What problem does the generation number in the file handle solve?

<showhide>
File names require path traversal on every operation, which is expensive and can race with concurrent renames. File handles encode the disk location (i-number) directly, making lookups O(1). The generation number solves i-number reuse: when a file is deleted and a new file is created reusing the same i-number, the old generation number no longer matches, so the server returns ESTALE and the client knows the original file is gone.

## NFS Consistency and Caching

NFS clients cache file data and attributes locally to reduce network traffic. This introduces consistency challenges: two clients may see different versions of the same file. NFS uses **close-to-open consistency**: when a client opens a file, it checks the server for a fresh copy; when it closes a file, it flushes dirty data to the server. Between open and close, a client may read stale data.

NFS also requires **synchronous writes** for metadata (directory updates, file creation) to ensure server-side durability—a client must not report success to an application until the data is on the server's stable storage. This is a major performance bottleneck for write-intensive workloads.

What is "close-to-open consistency" in NFS, and why is it weaker than the consistency you would get from a local disk?

<showhide>
Close-to-open consistency guarantees that when client B opens a file after client A has closed it, B sees all of A's writes. But there is no guarantee of consistency while both files are open simultaneously—A and B may have different cached versions of the same file with no mechanism to reconcile them until one closes. On a local disk, all reads and writes go through the same kernel buffer cache, so consistency is immediate.
