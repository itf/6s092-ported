# Crash Recovery for Disk File Systems

Crash recovery is a central theme in building reliable storage systems. A file system maintains complex on-disk data structures—i-nodes, directory blocks, free bitmaps, indirect block pointers—that must remain internally consistent even if the machine crashes mid-operation. A single `rename` may require updating two directory blocks, an i-node, and a link count: a crash between any two of these updates leaves the disk in an inconsistent state.

The goals of crash recovery are: (1) maintain the storage system's internal invariants after a crash, and (2) preserve an ordered prefix of the user's operations—all completed operations should be reflected, and partial operations should be either completed or rolled back cleanly.

The lecture examines the FFS (Fast File System) approach of synchronous metadata writes, its shortcomings, and the motivation for logging (covered in the next lecture).

## FFS: Synchronous Metadata Updates

FFS (Berkeley Fast File System) handles crash recovery by carefully ordering synchronous writes to disk. Before allocating a block to a file, the block is zeroed; before adding a directory entry, the i-node is written. This ordering ensures that recovery (via `fsck`) can detect and repair most inconsistencies by scanning the entire disk.

However, `fsck` is slow (minutes to hours for large disks), FFS is slow (synchronous writes stall the CPU waiting for disk), and some operations (like `rename`) cannot be made safe with a single committing write—the ordering of two directory updates cannot be collapsed into one atomic write.

<checkyourself>
Why does FFS's approach of ordering synchronous writes fail to handle the `rename` operation correctly?

<showhide>
`rename(d2/f2, d1/f1)` must update two directory blocks: remove the entry from `d2` and add it to `d1`. These blocks are typically different disk sectors. There is no single "committing write" that atomically updates both. If the machine crashes after writing `d1` but before writing `d2`, the file appears in both directories (link count mismatch). If it crashes after writing `d2` but before writing `d1`, the file is lost. `fsck` can detect the inconsistency but cannot determine which directory entry was the intended final state.
</showhide>

## Commit Points and Ordering

The general framework for crash-safe updates: (1) each operation takes storage from one legal state to another, (2) updates are ordered to disk so there is a clear **commit point** (one write that atomically transitions from "not done" to "done"), and (3) recovery can finish or undo any partial operation by examining the state relative to the commit point.

For a two-step operation like `rename`, the standard technique is to use a **write-ahead log**: write a record describing the intended operation to a log on disk first (the commit point), then apply the changes. If a crash occurs after the log record is written, recovery can replay it. If a crash occurs before the log record is complete, recovery ignores the partial record.

What is the "shadow paging" (copy-on-write) approach to crash recovery, and what is its main drawback compared to logging?

<showhide>
Shadow paging prepares a new copy of all affected disk blocks, then atomically swaps in the new tree root with a single write. If the machine crashes before the root swap, the old tree is intact; after the swap, the new tree is visible. The drawback is write amplification: even a small update requires copying the entire path from the modified block to the root (all ancestor indirect blocks and the superblock), which is expensive for deep trees and large files.
</showhide>
