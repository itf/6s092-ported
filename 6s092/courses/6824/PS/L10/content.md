# Memory Consistency (Part 2)

This lecture continues the study of memory consistency models, examining weaker models that allow more hardware and compiler optimizations at the cost of more complex programmer reasoning. The key insight is that **sequential consistency is not required for correct programs**—programmers can write correct parallel programs under weaker models if they use synchronization operations (locks, barriers) at the right places.

The lecture introduces **release consistency** (RC), used in the TreadMarks distributed shared memory system. RC separates memory accesses into ordinary reads/writes and **synchronization operations** (acquire/release). Consistency is only required at synchronization points: before an acquire, all writes by the previous lock holder are guaranteed to be visible.

This maps directly to distributed systems: Ivy (a distributed shared memory system), TreadMarks, and the ccfs lab all use variants of this idea.

## Relaxed Consistency Models

Weaker than SC: **total store order (TSO)** allows reads to return locally-buffered (not yet globally visible) values. **Processor consistency** further relaxes ordering. **Weak consistency** only requires that all processors agree on the order of synchronization operations, not ordinary memory accesses.

The key observation: if a program has **no data races** (all accesses to shared variables are protected by locks), then any execution under a weak model produces the same result as some SC execution. Data-race-free programs are SC-equivalent under most weak models. This is the "DRF guarantee."

<checkyourself>
What is the "data-race-free" (DRF) guarantee, and why does it allow programmers to reason about programs under weak consistency models as if they were sequentially consistent?

<showhide>
A program is data-race-free if every pair of conflicting accesses (to the same variable, at least one is a write, by different threads) is separated by a synchronization edge (e.g., a lock acquire/release). The DRF guarantee states: if a program is data-race-free, any execution on a weak model produces only results that are possible under SC. This allows programmers to assume SC when reasoning about their program, as long as they correctly synchronize all shared accesses with locks or barriers.

## TreadMarks and Lazy Release Consistency

**TreadMarks** implements distributed shared memory (DSM) over a network of workstations using **lazy release consistency (LRC)**. Under LRC, writes are not propagated at release time; instead, they are propagated lazily to a processor only when it acquires the lock and requests the data it needs.

TreadMarks uses **diffs** to represent the changes a processor made to a page: when another processor needs the page, TreadMarks applies the accumulated diffs. **Vector timestamps** (covered in Lecture 12) track which writes each processor has seen, ensuring the correct diffs are applied.

What is "false sharing" in distributed shared memory, and why does it reduce the performance benefit of LRC?

<showhide>
False sharing occurs when two processors frequently access different variables that happen to reside on the same page (the unit of consistency in TreadMarks). Even though they never share data, they contend on the same page—each write by one processor invalidates the other processor's cached copy, forcing a diff fetch on next access. The page granularity is much coarser than the actual sharing granularity, so LRC must propagate more data than necessary. Solutions include using smaller pages, restructuring data to separate frequently-written variables, or using object-based DSM with finer granularity.
</showhide>
