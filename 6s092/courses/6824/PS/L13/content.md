# Paxos

Paxos (Lamport, 1989/2001) is the foundational algorithm for **consensus** in the presence of failures. While 2PC requires all participants and the TC to be up, Paxos can make progress as long as a **majority** of servers are available. This makes it much more fault-tolerant.

The key application is **state machine replication**: any deterministic service (a lock server, a storage system, a database) can be replicated across multiple servers by ensuring all replicas see the same sequence of client operations in the same order. Paxos is used to agree on each operation in the log.

The core difficulty: if the primary crashes, you need to elect a new one—but you must be certain the old primary is not still active (two primaries simultaneously would violate consistency). Paxos solves this without any external oracle, using only majority quorums.

## Paxos Overview: Single-Value Consensus

Paxos solves the problem of getting $2f+1$ servers to agree on a single value, tolerating up to $f$ failures. The protocol has two phases run by a **proposer**:

**Phase 1 (Prepare):** Proposer sends `PREPARE(n)` with a unique proposal number $n$ to a majority of servers (**acceptors**). Each acceptor replies with a promise not to accept proposals numbered less than $n$, and reports the highest-numbered proposal it has already accepted (if any).

**Phase 2 (Accept):** If the proposer receives promises from a majority, it sends `ACCEPT(n, v)` where $v$ is the value of the highest-numbered previously accepted proposal (if any) or the proposer's own value. Acceptors accept if they have not promised a higher number. Once a majority accepts, the value is **chosen**.

<checkyourself>
Why must the proposer in Phase 2 use the value from the highest-numbered previously accepted proposal (returned in Phase 1 replies), rather than its own preferred value?

<showhide>
A previous proposer may have already gotten a value accepted by a majority—i.e., the value is already "chosen" even though this proposer doesn't know it. If this proposer used a different value, it might try to choose a different value from what was already chosen, violating agreement. By always using the highest-numbered previously accepted value, Paxos ensures that once a value is chosen, all future proposals converge to the same value.
</showhide>

## Paxos for Multi-Value Log (Multi-Paxos)

Single-value Paxos is extended to **Multi-Paxos** for state machine replication: run one Paxos instance per log slot. Optimization: once a leader is elected (Phase 1 accepted by a majority for all future slots), it can skip Phase 1 for subsequent proposals and go directly to Phase 2 for each new client request. This reduces the common-case message count to 2 round trips per log entry.

What is the "livelock" problem in Paxos (also called dueling proposers), and how does the Paxos paper recommend avoiding it?

<showhide>
If two proposers repeatedly issue competing PREPARE requests with increasing proposal numbers, each can interrupt the other's Phase 2 before a value is chosen. Proposer A sends PREPARE(1), gets promises; proposer B sends PREPARE(2), gets promises and invalidates A's; A sends PREPARE(3), invalidates B's; and so on—neither ever completes Phase 2. Lamport's recommendation: use a distinguished leader (one proposer at a time), elected by some external mechanism (often Paxos itself or a timeout-based leader election). With a single leader, there is no competing proposer to cause livelock.
</showhide>
</checkyourself>
