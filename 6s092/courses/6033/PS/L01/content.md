# Modularity via Client/Server Organization

Building large, complex systems requires principled strategies for managing complexity. The first strategy is **modularity**: decomposing a system into components with well-defined interfaces so that each component can be reasoned about independently. A module's internals can change freely as long as its interface is preserved—this is the key benefit of abstraction.

Even modular systems can fail in hard-to-debug ways when modules share memory. A buggy module can corrupt another module's memory, causing mysterious failures far from the root cause. **Enforced modularity** prevents these interactions by restricting what modules can access. Client/server organization is one realization of enforced modularity: clients and servers communicate only through explicit messages, never through shared memory.

The client/server model maps naturally onto distributed systems—clients and servers may run on different machines connected by a network—but the concepts apply equally to modules on a single machine. The price of enforced modularity is communication overhead: instead of a function call, interactions require marshaling arguments into messages and sending them across an interface boundary.

## Complexity and Modularity

Modern software systems are enormously complex. Emergent behavior—where the whole system behaves in ways not easily predicted from any single component—makes debugging and reasoning hard. The goal of modularity is to limit **fate sharing**: if one component fails, others should be able to continue.

What property of client/server communication prevents a faulty client from corrupting server state in ways that a simple function call would not prevent?

<showhide>
In a client/server model, the client cannot directly access server memory. All interaction is through explicit messages. The server validates each request before acting on it. A buggy or malicious client can only cause harm that the server's request-handling code explicitly allows—the server's address space is protected. With a direct function call and shared memory, the caller could corrupt any data the callee can access.

## Abstraction and Interface Design

A good interface hides implementation details while exposing exactly what clients need. The **abstraction** lets the implementation evolve (e.g., replacing a database backend) without changing clients. The key rule: clients should not depend on implementation details not specified in the interface.

<checkyourself>
Why is it important for a module's interface to be simpler than its implementation?

<showhide>
If the interface is as complex as the implementation, clients cannot understand how to use the module without understanding its internals—defeating the purpose of abstraction. A simpler interface means clients need to know less, reducing the surface for misuse, and gives implementors more freedom to change internals. The reduction in what clients must know is the core value of abstraction.
