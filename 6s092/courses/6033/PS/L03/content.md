# Operating Systems Structure and Virtual Memory

Operating systems enforce modularity on a single machine by giving each program the illusion that it owns the entire machine. The key OS abstractions are **processes** (isolated execution environments), **virtual memory** (each process has its own address space), and **system calls** (the controlled interface from user programs into the kernel).

**Virtual memory** is a naming system: virtual addresses are names, physical memory locations are values, and the page table is the lookup structure. Each process has its own page table, so the same virtual address in two different processes maps to different physical memory—complete isolation. The OS (with hardware help from the MMU) translates every memory access at runtime.

Binding is the act of associating a name with a value. In virtual memory, binding happens at different times: **compile time** (hard-coded addresses), **load time** (the linker resolves symbols), or **run time** (the MMU translates on every access). Late binding (run time) is most flexible and enables features like copy-on-write, demand paging, and memory-mapped files.

## Virtual Address Translation

The page table maps virtual page numbers to physical frame numbers. On a 32-bit system with 4 KB pages, the virtual address space has $2^{32} / 2^{12} = 2^{20}$ pages. A naive single-level page table would require $2^{20}$ entries × 4 bytes = 4 MB per process—large but manageable. Multi-level page tables reduce this by only allocating entries for pages that are actually used.

Why does each process have its own page table, and what security property does this enforce?

<showhide>
Each process has its own page table so that virtual addresses in one process map to different physical frames than the same virtual addresses in another process. This means one process cannot read or write another process's memory simply by using its virtual addresses—the MMU will translate them to different (or invalid) physical locations. This is **memory isolation**, a core security property enforced by the OS and hardware.

## OS Abstractions and System Calls

User programs run in **user mode** and cannot execute privileged instructions (e.g., directly accessing I/O ports, modifying page tables). To request OS services, they execute a **system call**, which switches to **kernel mode** and transfers control to a kernel entry point. The kernel validates the request and performs the operation on behalf of the process.

What is demand paging and why does it allow running programs larger than physical memory?

<showhide>
Demand paging loads pages from disk into physical memory only when they are first accessed (i.e., when a page fault occurs). Pages that are not accessed are never loaded; infrequently used pages can be evicted to disk when physical memory is needed. This means the sum of all virtual address spaces can far exceed physical memory—only the **working set** (pages actively in use) needs to be resident at any given time.
