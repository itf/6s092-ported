# Isolation via Serializability

Isolation ensures that concurrent transactions produce the same result as some serial execution. The gold standard is **serializability**: the execution is equivalent to some serial order of the transactions. If we can find a serial order that produces the same result as the actual concurrent execution, the execution is serializable and correct.

**Conflict serializability** is a practical criterion. Two operations **conflict** if they access the same data item and at least one is a write. A schedule is conflict-serializable if its **conflict graph** (with a node per transaction and an edge from Ti to Tj for each conflict where Ti precedes Tj) is **acyclic**. An acyclic conflict graph means we can topologically sort transactions into a valid serial order.

**Two-phase locking (2PL)** is the classic protocol for achieving serializability. A transaction must acquire a lock on data before accessing it (growing phase) and must not acquire new locks after releasing any lock (shrinking phase). 2PL guarantees conflict-serializable schedules but can cause deadlock.

## Conflict Graphs

Conflicts are the key: read-write, write-read, and write-write operations on the same object constitute conflicts. Read-read is not a conflict (both are read-only, order doesn't matter). The conflict graph captures ordering constraints: if T1 writes X before T2 reads X, there's an edge T1→T2 in the conflict graph.

<checkyourself>
Explain why a cycle in the conflict graph means the schedule is not serializable.

<showhide>
A cycle T1 → T2 → T1 means: T1 must come before T2 (due to a conflict where T1 precedes T2), AND T2 must come before T1 (due to another conflict). These constraints are contradictory—no serial order satisfies both. Therefore, no serial execution is equivalent to this schedule, and the schedule is not serializable. An acyclic graph allows a topological sort, which gives a valid serial order.

## Two-Phase Locking

2PL has two phases: **growing** (acquire locks, never release) and **shrinking** (release locks, never acquire). The transaction can access data freely while holding locks. At commit/abort, all locks are released. The lock point (moment all needed locks are held) determines the serial order.

What is strict two-phase locking (S2PL) and why is it used in practice over basic 2PL?

<showhide>
In basic 2PL, a transaction can release locks before committing (as long as it doesn't acquire new ones). This can lead to **cascading aborts**: if T1 releases a lock on X and T2 reads X, then T1 aborts, T2 has read a "dirty" (uncommitted) value and must also abort. Strict 2PL (S2PL) holds all locks until the transaction commits or aborts. This prevents cascading aborts (no transaction reads uncommitted data) at the cost of holding locks longer, increasing contention. Most real systems use S2PL or a variant.
</showhide>
