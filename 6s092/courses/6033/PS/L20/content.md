# Introduction to Security

Security in computer systems requires a different mindset from reliability. In reliability, we assume components fail independently and randomly. In security, we face an **adversary** who is intelligent, adaptive, and actively trying to subvert the system. We must assume the adversary knows the system design (Kerckhoffs' principle) and will exploit any weakness.

The **security design process** starts with a **threat model**: who are the adversaries? What are their capabilities (can they read network traffic? modify messages? compromise individual hosts?)? What are their goals? A security mechanism that is effective against one adversary may be useless against a more powerful one.

**Policy** defines what we want to protect: confidentiality (information only reaches authorized parties), integrity (data is not modified by unauthorized parties), and availability (authorized users can access resources). The **guard model** structures access control: a guard stands between principals and resources, enforces policy by checking permissions before granting access, and maintains an audit log.

## Threat Models

A threat model explicitly states assumptions about the adversary. Without a threat model, security analysis is meaningless: "secure" against what? Common attacker capabilities include: passive eavesdropping (wiretapping), active tampering (modifying messages), replay attacks (re-sending captured messages), compromising endpoints (malware on the client).

Why is having an explicit threat model essential for evaluating security?

<showhide>
Without a threat model, you cannot determine whether a security mechanism provides meaningful protection. For example, encrypting disk data protects against an attacker who steals the disk but does nothing against an attacker who compromises the running OS. HTTPS protects against network eavesdroppers but not against malware on the client's machine. An explicit threat model defines what you're protecting against, allowing you to say "this mechanism is secure against adversary X with capabilities Y"—and to recognize when an assumed threat is not covered.

## The Guard Model

The guard model structures access control: a **reference monitor** (guard) intercepts every access request, checks the **access control list (ACL)** or **capability** for the requesting principal and requested resource, and grants or denies access. For complete mediation, EVERY request must go through the guard—no bypass paths.

What is the principle of least privilege, and give an example of how violating it increases security risk?

<showhide>
**Least privilege**: a component should have only the minimum permissions needed to perform its function. Example of violation: a web server runs as root. A buffer overflow vulnerability in the web server gives the attacker root access to the entire system. If the web server ran as a restricted user with access only to the web root directory, the attacker would gain only those limited permissions. Least privilege limits the blast radius of a compromise by ensuring that a compromised component cannot access resources beyond its legitimate function.
</showhide>
