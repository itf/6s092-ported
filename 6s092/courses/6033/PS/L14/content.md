# Fault Tolerance: Reliability via Replication

Real systems fail. Hardware fails, software has bugs, operators make mistakes, and power goes out. The goal of fault-tolerant systems is to maintain correct behavior despite these failures. The key technique is **replication**: store multiple copies of data or run multiple copies of services so that failures of individual components do not cause data loss or unavailability.

**RAID (Redundant Array of Independent Disks)** is a classic example of fault tolerance via replication. RAID 1 mirrors data across two disks: if one fails, the other has a complete copy. RAID 5 uses parity across N disks: it can reconstruct data after any single disk failure using XOR, with only 1/N storage overhead (vs. 2× for RAID 1).

Building reliable systems requires defining the **failure model**: what kinds of failures can occur? Common models include **fail-stop** (a component either works correctly or stops completely), **crash-recovery** (a component can crash and restart with durable state), and **Byzantine** (a component can behave arbitrarily, including lying). The failure model determines which fault-tolerance techniques apply.

## RAID and Storage Reliability

RAID 5 distributes parity across all disks. With N disks, RAID 5 has N-1 disks of usable capacity and survives any single disk failure. After a failure, the array is in a **degraded** state—it can still serve reads (by computing missing data from parity) but is at risk: a second disk failure would cause data loss. Reconstruction is triggered immediately.

<checkyourself>
How does RAID 5 reconstruct the data on a failed disk using XOR parity?

<showhide>
RAID 5 stores the XOR of corresponding blocks across all disks in a parity block. For disks D1, D2, D3, D4, the parity P = D1 ⊕ D2 ⊕ D3 ⊕ D4. If D3 fails, D3 = D1 ⊕ D2 ⊕ D4 ⊕ P (XOR all other disks including parity). This works because XOR is its own inverse: X ⊕ X = 0, so XOR-ing all known values recovers the missing one. Reconstruction reads all other disks and XORs them together.

## Replication and Availability

For higher availability, systems replicate across failure domains (different racks, data centers, geographic regions). **Replication factor** determines how many failures can be tolerated: with 3 replicas, the system survives 2 simultaneous failures. But replication requires keeping replicas consistent—a harder problem than local RAID.

What is the difference between high availability and durability, and why must a system be designed to achieve both separately?

<showhide>
**Availability** means the system is accessible and responsive when needed. **Durability** means stored data is not lost even after failures. A system can have high availability but low durability (e.g., an in-memory cache that's always up but loses data on restart) or high durability but low availability (e.g., a tape archive that never loses data but takes hours to restore). RAID provides durability by surviving disk failures; database replication with failover provides availability by automatically switching to a replica. Most production systems require both, achieved through separate techniques.
</showhide>
