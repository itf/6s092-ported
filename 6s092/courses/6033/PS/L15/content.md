# Fault Tolerance: Introduction to Transactions

Replication is not enough for all failure scenarios. Consider a bank transfer: debit account A, credit account B. If the system crashes between the debit and credit, A is debited but B is never credited—money is lost. Replication doesn't help here; we need **atomicity**: either both operations complete or neither does.

A **transaction** groups multiple operations into an all-or-nothing unit. Transactions have the **ACID** properties: **Atomicity** (all or nothing), **Consistency** (the database moves from one valid state to another), **Isolation** (concurrent transactions don't interfere), and **Durability** (committed transactions survive crashes). Atomicity and durability together address the failure scenario above.

**Shadow copies** are a simple implementation of atomicity: keep the current data untouched while writing updates to a shadow copy, then atomically switch a pointer from the old version to the new. If the system crashes during the update, the old version is still valid; if it crashes after the pointer switch, the new version is fully committed.

## Atomicity via Shadow Copies

Shadow copy atomicity works when "switching the pointer" is itself an atomic operation. On Unix, renaming a file within the same filesystem is atomic (guaranteed by the OS): the directory entry atomically switches from the old name to the new name. Shadow copies exploit this: write the new version to a temp file, then rename it over the old file.

<checkyourself>
What is the failure-atomicity guarantee provided by shadow copies, and what are its limitations?

<showhide>
Shadow copies guarantee that after any crash, the database is in either the old state (rename didn't happen) or the new state (rename completed). There is no partial state. Limitation: shadow copies don't compose—if a transaction updates multiple files, renaming each one is not atomic as a group. Also, shadow copies require copying the entire data structure even for small updates (expensive for large databases). Logging (WAL) addresses both limitations.

## Isolation and Transactions

**Isolation** ensures that concurrent transactions produce the same result as if they ran serially. Without isolation, two concurrent transactions can interfere: a read-modify-write cycle can produce incorrect results if another transaction modifies the same data between the read and the write.

What is the difference between atomicity and durability, and why do you need both for correct transaction semantics?

<showhide>
**Atomicity** guarantees all-or-nothing: either all operations in the transaction happen or none do. It protects against partial execution due to crashes or errors. **Durability** guarantees that once a transaction is committed, its effects survive subsequent crashes. You need both: atomicity without durability means a committed transaction can be lost in a crash; durability without atomicity means a partial transaction (some operations done, some not) can be permanently stored after a crash. Together, they ensure transactions either fully commit (and persist) or fully abort (leaving no trace).
</showhide>
