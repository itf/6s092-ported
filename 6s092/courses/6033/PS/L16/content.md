# Atomicity via Logging

Shadow copies are impractical for large databases: copying the entire database for every transaction is too slow. **Write-ahead logging (WAL)** achieves atomicity efficiently by recording changes in a log before applying them to the database. The log is the ground truth; the database is derived from it.

WAL works as follows: before modifying any data, write a log record describing the intended change. On commit, write a COMMIT record to the log and flush it to durable storage. On recovery after a crash, replay the log to reconstruct committed transactions and undo any transactions that were in-progress at the time of the crash.

The key insight is that log records are **append-only** (sequential writes, fast) while database updates are random-access. By deferring or batching database updates and writing the log first (the WAL property), the system achieves high throughput while guaranteeing atomicity and durability.

## Write-Ahead Logging

The log records contain: transaction ID, operation (write/update), data location, old value (for undo), and new value (for redo). **UNDO** rules allow rolling back uncommitted transactions. **REDO** rules allow reapplying committed transactions after a crash. Together, they allow the database to reach any committed state.

<checkyourself>
Why must the log record be written to durable storage BEFORE the database page is modified (the "write-ahead" property)?

<showhide>
If the database page were modified first and the system crashed before writing the log record, there would be no record of what changed. The database would be in an inconsistent state with no way to determine whether the change was part of a committed transaction or not. Writing the log first ensures that on recovery, the log always has enough information to either redo (if committed) or undo (if not committed) every change. The log record must be durable before the data page, so a flush (fsync) of the log is required before modifying data.
</showhide>

## Log-Structured Storage

Logs enable efficient recovery but also inspire the **log-structured** storage approach: treat the entire storage system as an append-only log. All writes append new versions; reads find the latest version by scanning (or via an index). This turns random writes into sequential writes, dramatically improving write throughput on spinning disks and SSDs. Periodic compaction reclaims space from superseded entries.

What is the role of checkpoints in WAL, and how do they reduce recovery time?

<showhide>
Without checkpoints, recovery must replay the entire log from the beginning—potentially hours of log for a long-running system. A **checkpoint** is a point in the log where the system ensures all committed changes have been flushed to the database. After a checkpoint, the log before that point can be truncated: recovery only needs to replay log records after the most recent checkpoint. This bounds recovery time to the amount of work done since the last checkpoint, regardless of how long the system has been running.
</showhide>
