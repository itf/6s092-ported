# Secure Channels

Communicating over an untrusted network requires cryptographic protection. A **secure channel** provides confidentiality (eavesdroppers cannot read messages), integrity (tampering is detected), and authentication (both parties know who they're talking to). TLS (Transport Layer Security) is the protocol that provides secure channels for the web.

**Symmetric encryption** (e.g., AES) uses the same key for encryption and decryption—fast but requires a shared key. **Asymmetric (public-key) encryption** (e.g., RSA) uses a key pair: messages encrypted with the public key can only be decrypted with the private key. Asymmetric cryptography solves the key distribution problem but is orders of magnitude slower than symmetric.

**Digital signatures** use asymmetric keys in reverse: a message is signed with the **private key** and verified with the **public key**. Signatures provide authentication and non-repudiation: only the holder of the private key could have produced the signature. **Message Authentication Codes (MACs)** provide integrity and authentication using a shared symmetric key but not non-repudiation.

## TLS Handshake

TLS uses asymmetric cryptography for **key exchange** (establishing a shared symmetric key) and then switches to symmetric encryption for bulk data. The server presents a **certificate** (public key signed by a Certificate Authority) that proves its identity. The client verifies the certificate, uses the server's public key to securely exchange a session key, and then all data is encrypted with the session key.

Why does TLS use asymmetric cryptography only for the handshake and then switch to symmetric encryption for data transfer?

<showhide>
Asymmetric encryption (RSA, DH) is computationally expensive—hundreds to thousands of times slower than symmetric encryption (AES). It would be impractical to encrypt a large file transfer with RSA. TLS uses asymmetric cryptography once, during the handshake, to securely establish a shared session key. From then on, all data is encrypted with symmetric AES, which is fast enough for bulk data. This hybrid approach gets the best of both: asymmetric cryptography solves the key distribution problem; symmetric cryptography provides performance.

## Certificate Authorities

Asymmetric cryptography requires knowing the other party's authentic public key. **Certificate Authorities (CAs)** solve this: a CA signs a certificate binding a public key to an identity (e.g., "this public key belongs to mit.edu"). Clients trust a set of root CAs (pre-installed in browsers/OSes). Verifying a certificate chains to a trusted root: CA signs → intermediate CA signs → server certificate.

What is a man-in-the-middle (MITM) attack, and how does TLS with certificate verification prevent it?

<showhide>
In a MITM attack, an attacker intercepts communication between Alice and Bob. Alice sends messages to the attacker believing she's talking to Bob; the attacker forwards (possibly modified) messages to Bob. TLS prevents this through **certificate verification**: when Alice connects to Bob's server, it presents a certificate signed by a trusted CA. Alice's browser verifies that (1) the certificate is signed by a trusted CA, (2) the certificate names match the hostname Alice is connecting to, and (3) the certificate is not expired or revoked. If any check fails, the browser warns the user. The attacker cannot forge a valid certificate without the CA's private key.
