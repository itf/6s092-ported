# Anonymity and Bitcoin

Traditional internet protocols (TCP/IP) reveal source IP addresses, making anonymity difficult. Many legitimate reasons exist for anonymity: whistleblowers, journalists, political dissidents, privacy advocates. **Tor (The Onion Router)** provides anonymity by routing traffic through a chain of volunteer relays, encrypting the path so no single relay knows both the source and destination.

**Bitcoin** is a decentralized digital currency that solves the double-spending problem without a trusted central authority. It uses a **blockchain**—a distributed append-only ledger maintained by a peer-to-peer network. Consensus on the ledger is achieved via **proof-of-work**: nodes (miners) compete to solve computational puzzles; the longest chain is accepted as the authoritative ledger.

Both Tor and Bitcoin represent attempts to build systems with strong security properties (anonymity, decentralized trust) by encoding them into the cryptographic structure of the protocol rather than relying on trusted parties. The key insight: if you can't trust anyone, build a system where trust is not required.

## Tor and Onion Routing

Tor routes traffic through 3 relays: a **guard** (entry), **middle**, and **exit** node. The client builds a circuit by negotiating session keys with each relay in sequence. Traffic is encrypted in three layers (onion encryption): the exit node decrypts the outermost layer and sees only the destination, not the source; the guard sees the source but not the destination; no single relay knows both.

<checkyourself>
Why does Tor use three relays rather than one, and what attack does the third relay (exit) not need to defend against?

<showhide>
With one relay: the relay knows both the source (client) and destination. Compromising one relay breaks anonymity. With three relays: the guard knows the client's IP but not the destination; the exit knows the destination but not the client's IP; the middle knows neither. An adversary must compromise the guard AND exit simultaneously to link source and destination. The exit node sees the destination in plaintext but does not see the source IP—it only sees the middle relay's IP. The exit does NOT need to protect against knowing the source (it can't know it with proper onion routing).

## Blockchain and Proof-of-Work

Bitcoin's blockchain is a linked list of blocks, each containing a batch of transactions and the hash of the previous block. **Proof-of-work**: to add a block, a miner must find a nonce such that SHA-256(block_header + nonce) < target. This takes ~10 minutes on average network-wide. The **longest chain** wins—if two miners find valid blocks simultaneously, the network eventually converges on the chain that gets extended first.

<checkyourself>
What is the double-spending problem in digital currency, and how does Bitcoin's blockchain solve it without a trusted central party?

<showhide>
Digital currency is just data; unlike physical cash, it can be copied. The double-spending problem: Alice could sign a transaction giving 1 BTC to Bob, then sign another transaction giving the same 1 BTC to Carol. Without a trusted party to maintain a single authoritative ledger, both transactions appear valid. Bitcoin solves this via a distributed public ledger (blockchain) and proof-of-work consensus. All transactions are broadcast publicly; miners include transactions in blocks only if the inputs haven't been spent. The longest chain (most cumulative proof-of-work) is the authoritative ledger. Creating a fraudulent alternative history requires more hash power than the rest of the network—economically infeasible for a decentralized, well-distributed network.
