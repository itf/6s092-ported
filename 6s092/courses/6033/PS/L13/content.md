# Content Distribution and New Internet Technologies

The internet was designed for point-to-point communication, but modern applications—file sharing, video streaming, VoIP—have different requirements. These applications have driven new designs: **BitTorrent** for peer-to-peer file sharing, **CDNs** for scalable content delivery, **VoIP/Skype** for real-time communication, and **adaptive bitrate streaming** for video.

**BitTorrent** distributes large files by having each downloader simultaneously upload to others. The file is split into pieces; clients download pieces from multiple peers and upload to others. **Tit-for-tat** incentivizes reciprocity: a client uploads to the peers that upload fastest to it, and periodically "optimistically unchokes" a random peer to discover new fast uploaders. This makes free-riding costly.

**Content Delivery Networks (CDNs)** cache content at geographically distributed servers (edge nodes) close to users, reducing latency and offloading origin servers. DNS-based redirection routes users to nearby edge nodes. CDNs are now responsible for a large fraction of all internet traffic.

## BitTorrent and Peer-to-Peer

In BitTorrent, the **tracker** (or DHT) helps peers find each other; actual data flows directly between peers. The **rarest-first** piece selection strategy ensures rare pieces are quickly replicated, improving resilience. Tit-for-tat addresses the free-rider problem by preferentially uploading to peers that reciprocate.

<checkyourself>
Why does BitTorrent use a "rarest-first" piece selection strategy rather than downloading pieces in sequential order?

<showhide>
Rarest-first ensures that pieces with few copies in the swarm are replicated quickly. If a very rare piece exists only on one peer and that peer leaves, the piece is permanently lost from the swarm. By prioritizing rare pieces, the swarm becomes more resilient: all pieces quickly reach multiple peers. Sequential downloading would create hotspots (many peers downloading the same early pieces) and leave rare pieces unreplicated until the very end.

## VoIP and Adaptive Streaming

VoIP (e.g., Skype) uses UDP rather than TCP because retransmission-based reliability introduces variable latency that is worse than occasional loss for real-time audio. A 20 ms jitter buffer at the receiver smooths out network jitter. **Adaptive bitrate (ABR) streaming** (e.g., DASH) adjusts video quality dynamically based on available bandwidth—downloading high-quality segments when bandwidth is ample and lower-quality when congested.

How does a DHT (Distributed Hash Table) replace the BitTorrent tracker, and what advantage does this provide?

<showhide>
A DHT distributes the tracker function across all peers. Each peer is responsible for storing information about a subset of torrents (based on a hash of the torrent's info-hash). To find peers for a torrent, a new peer queries the DHT by iteratively finding the node closest to the torrent's hash. This eliminates the central tracker as a single point of failure and censorship—there is no central server to take down. DHTs used in BitTorrent (e.g., Kademlia) find peers in $O(\\log N)$ hops for $N$ nodes.
</showhide>
