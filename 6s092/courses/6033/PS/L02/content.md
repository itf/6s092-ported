# Naming in Systems

**Naming** is the binding of a name to a value (or object). Names appear everywhere in systems: file paths, IP addresses, DNS hostnames, variable names, process IDs, and URLs are all examples. A naming system has three components: the **name space** (the set of valid names), the **value space** (the set of things names can refer to), and the **lookup algorithm** (how a name is resolved to a value).

Good naming design has enormous practical impact. Names should be location-independent (the same name should work regardless of where the object has moved), human-friendly, and efficiently resolvable. A flat name space scales poorly; hierarchical names (like DNS domain names or Unix file paths) allow delegation and decentralized management.

The Domain Name System (DNS) is the canonical example of naming in a large-scale distributed system. It maps human-readable hostnames to IP addresses using a hierarchical, delegated namespace managed by distributed servers. Understanding DNS illuminates general principles about caching, consistency, and the tradeoffs between freshness and performance.

## Name Resolution and DNS

DNS organizes names in a tree. The root (`.`) delegates to top-level domains (`.com`, `.edu`, `.io`), which delegate further. Each zone is managed by an **authoritative name server** for that zone. Resolvers walk the tree from root to leaf to answer queries, caching results to amortize the cost.

What is the role of caching in DNS, and what consistency problem does it create?

<showhide>
Caching reduces the load on authoritative servers and speeds up resolution: a resolver can answer a query from its cache without contacting any remote server. The consistency problem is **stale data**: if an authoritative record changes (e.g., a host's IP address changes), cached copies continue to return the old value until their TTL expires. DNS accepts this eventual consistency; critical applications may need to reduce TTL or use other mechanisms.

## Naming Tradeoffs: Indirection and Mobility

Indirection via naming is powerful: if an object moves, you update one mapping and all names continue to resolve correctly. Without indirection, every reference to the old location must be updated. DNS exploits this: changing a CNAME or A record redirects all clients without touching clients directly.

<checkyourself>
Why does DNS use a hierarchical namespace rather than a flat one?

<showhide>
A flat namespace requires a single authority to manage all name-to-value bindings, which does not scale to billions of names and cannot be managed by decentralized organizations. A hierarchy delegates sub-namespaces to sub-authorities: `.edu` is managed by Educause, `mit.edu` is managed by MIT, `csail.mit.edu` is managed by CSAIL. Each organization manages only its own zone, and the system scales to any size without a central bottleneck.
</showhide>
