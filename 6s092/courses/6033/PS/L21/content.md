# Authentication and Passwords

Authentication answers the question: is this principal who they claim to be? The classic mechanism is **passwords**: the user knows a secret; the server checks that the presented secret matches what it stored. Password authentication is ubiquitous but has well-known weaknesses: users choose weak passwords, reuse passwords, and passwords can be stolen from compromised databases.

To protect against database breaches, servers should store **salted hashes** rather than plaintext passwords. A **salt** is a random value appended to the password before hashing; it is stored alongside the hash. Salts prevent precomputed dictionary attacks (**rainbow tables**): an attacker who steals the database must crack each password individually.

**Multi-factor authentication (MFA)** combines something-you-know (password) with something-you-have (TOTP token, hardware key) or something-you-are (biometric). MFA significantly raises the bar for attackers: stealing a password database is no longer sufficient.

## Password Storage

Without salts, two users with the same password have the same hash. An attacker can precompute a rainbow table mapping common passwords to their hashes and look up all hashes in a stolen database instantly. With salts, each password is hashed with a unique random salt; the same password produces different hashes for different users. The attacker must crack each hash separately.

<checkyourself>
Why does salting prevent rainbow table attacks, and what attack does it NOT prevent?

<showhide>
Salting prevents rainbow tables because each password is combined with a unique random salt before hashing. A precomputed table of (hash → password) is useless: the attacker would need to precompute a separate table for every possible salt value. However, salting does NOT prevent **brute-force attacks**: the attacker can still try candidate passwords for a specific user by computing hash(candidate + salt) and comparing to the stored hash. Salting only prevents attacks that leverage precomputed tables. Using a slow hash (bcrypt, Argon2) mitigates brute force by making each hash computation expensive.

## Authentication Protocols

Even with correct password storage, the authentication protocol itself must be secure. Sending passwords in cleartext over the network allows eavesdroppers to steal them. **TLS** encrypts the channel so passwords are never transmitted in plaintext. **Challenge-response** protocols avoid sending the password at all: the server sends a random challenge; the client responds with HMAC(password, challenge), proving knowledge of the password without revealing it.

What is TOTP (Time-based One-Time Password) and how does it provide two-factor authentication without requiring server-side storage of the OTP?

<showhide>
TOTP (RFC 6238) generates a one-time password by computing HMAC(secret, current_time_step) where the time step is typically 30 seconds. The user's device and the server both know the shared secret; they independently compute the same OTP for the current time window. The server does not store OTPs—it recomputes the expected OTP from the shared secret when the user submits one. An attacker who intercepts an OTP can only use it within the 30-second window. Since each OTP is used at most once (server tracks used OTPs within the window), replay attacks are prevented.
</showhide>
