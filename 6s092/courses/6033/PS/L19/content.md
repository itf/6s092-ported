# Replicated State Machines and Availability

2PC requires all participants to be available to commit. For high availability under failures, we need protocols that can make progress even when some nodes fail. **Replicated State Machines (RSM)** achieve this: a set of servers execute the same sequence of commands and maintain identical state. As long as a **quorum** (majority) of servers is available, the system makes progress.

**Paxos** and its variants (Multi-Paxos, Raft) are consensus protocols that allow a set of nodes to agree on a sequence of values (the log of commands) even when some nodes crash. Paxos tolerates $f$ failures with $2f+1$ nodes: a majority ($f+1$ nodes) is always available even if $f$ fail.

The **CAP theorem** captures a fundamental tradeoff: a distributed system cannot simultaneously provide **Consistency** (all reads see the latest write), **Availability** (every request gets a response), and **Partition tolerance** (the system works despite network partitions). You must choose CP (consistent but unavailable during partitions) or AP (available but possibly stale during partitions).

## Replicated State Machines

An RSM processes client requests by running a consensus protocol on each request before executing it. The consensus protocol determines the order of requests. Once a request is committed to the log (accepted by a quorum), all replicas apply it in the same order, maintaining identical state.

How does Paxos ensure that only one value is chosen even when multiple proposers are active simultaneously?

<showhide>
Paxos uses **two phases** and **ballot numbers**. In Phase 1 (Prepare), a proposer gets acceptors to promise not to accept lower-numbered proposals. In Phase 2 (Accept), the proposer sends its value to acceptors, but only if it has heard from a quorum. If two proposers race, the one with the higher ballot number wins (the other's Phase 2 will be rejected by acceptors that promised to ignore lower ballots). The quorum intersection property guarantees that any two quorums overlap: the second proposer must hear from at least one acceptor that participated in the first quorum, learning any previously chosen value.

## CAP Theorem

The CAP theorem (Brewer 2000, formalized by Gilbert & Lynch 2002) states that in the presence of a network partition, a system must choose between consistency and availability. Most real systems choose one of: CP (e.g., Zookeeper, etcd—refuse writes during partitions to stay consistent) or AP (e.g., Cassandra with eventual consistency—accept writes from both sides of a partition, reconcile later).

What is the difference between strong consistency (linearizability) and eventual consistency, and when is each appropriate?

<showhide>
**Linearizability** (strong consistency): every operation appears to take effect at a single, well-defined point in time between its start and end. A read always sees the latest committed write. This requires coordination (quorum reads/writes) and sacrifices availability during partitions. Appropriate for financial systems, distributed locks, configuration management. **Eventual consistency**: after updates stop, all replicas eventually converge to the same value. Reads may return stale data. No coordination required; high availability. Appropriate for social media feeds, shopping carts, DNS—where stale reads are acceptable and high availability is critical.
</showhide>
