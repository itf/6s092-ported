# Threads

A **thread** is the unit of concurrency within a process. Multiple threads share a process's address space, file descriptors, and other resources, but each has its own stack and program counter. Threads enable overlapping computation with I/O: while one thread waits for a disk read, another can execute.

**Condition variables** are synchronization objects used with locks to allow threads to wait for a condition to become true. The key operations are `wait(cv, lock)`—which atomically releases the lock and suspends the thread—and `signal(cv)`—which wakes one waiting thread. `broadcast(cv)` wakes all waiters. Condition variables must always be used inside a while loop (not if) to handle **spurious wakeups** and the TOCTOU gap between when a condition is checked and when the lock is reacquired.

**Preemption** allows the OS scheduler to forcibly context-switch a running thread, giving the CPU to another thread. Without preemption, a thread that never yields (e.g., an infinite compute loop) would monopolize the CPU. With preemption, the OS can enforce fair sharing via timer interrupts.

## Condition Variables

Condition variables decouple the lock (which enforces mutual exclusion) from waiting for a condition. The classic pattern: acquire lock, while condition is false: wait on CV (atomically releases lock and sleeps), re-check condition, then proceed.

<showhide>

The while loop re-checks the condition after reacquiring the lock, ensuring the thread only proceeds when the condition is actually true.

## Thread Scheduling and Preemption

The OS scheduler decides which thread runs on each CPU. **Preemption** via timer interrupts means a thread can be interrupted at any instruction boundary. This is what makes the lock/unlock discipline essential: if a thread is preempted while holding a lock, other threads trying to acquire the lock will block, but the system still makes progress once the preempted thread is rescheduled and releases the lock.

What is the difference between signal() and broadcast() on a condition variable, and when should you use each?

<showhide>
`signal()` wakes exactly one waiting thread; `broadcast()` wakes all waiting threads. Use `signal()` when exactly one waiting thread can make progress (e.g., one slot opened in a full buffer—only one producer can insert). Use `broadcast()` when multiple threads might be able to make progress (e.g., a reader-writer lock released for reading—all waiting readers can proceed), or when you're not sure which waiter should be woken. Using `signal()` incorrectly can leave threads stuck even when the condition they're waiting for is satisfied.
</showhide>
