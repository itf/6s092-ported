# Operating System Structure and Virtual Machines

There are two major OS kernel architectures: **monolithic kernels** and **microkernels**. A monolithic kernel runs all OS services (file system, device drivers, network stack, memory manager) in kernel mode as a single large program. A microkernel moves most services into user-space servers, keeping only the minimal mechanisms (IPC, virtual memory, basic scheduling) in the kernel.

**Virtual machines (VMs)** push the idea of enforced modularity further: a **hypervisor** (VMM) runs directly on hardware and presents each VM with a virtual machine interface that looks like real hardware. Each VM runs an unmodified OS. VMs provide even stronger isolation than processes: a crash in one VM cannot affect another, and VMs can run different operating systems on the same physical host.

The tradeoff between monolithic and microkernel architectures is a classic systems debate. Monolithic kernels are fast (services call each other directly in kernel memory) but risky (a bug in any service can crash the kernel). Microkernels have better fault isolation (a failing driver is just a failed user-space process) but higher IPC overhead.

## Monolithic vs. Microkernel

In a monolithic kernel, all OS services share kernel address space. A bug in a device driver can corrupt kernel data and crash the whole machine. Linux is a prominent monolithic kernel. In a microkernel (e.g., L4, Mach), the kernel is tiny; drivers run as user processes communicating via IPC.

What is the main reliability advantage of a microkernel over a monolithic kernel, and what is the main performance disadvantage?

<showhide>
**Reliability**: A faulty device driver running as a user-space server can only corrupt its own address space. The microkernel can kill and restart the driver process without affecting the rest of the system. In a monolithic kernel, the same bug would corrupt kernel memory and likely crash the machine. **Performance**: Service-to-service calls in a monolithic kernel are cheap function calls within kernel space. In a microkernel, the same calls require IPC (message passing through the kernel), which involves context switches, message copying, and scheduling overhead—typically 5-100× more expensive.

## Virtual Machine Implementation

The hypervisor must handle **privileged instructions** that a guest OS tries to execute. On hardware that supports virtualization (e.g., Intel VT-x), privileged instructions automatically trap to the hypervisor. The hypervisor emulates the instruction's effect on the virtual machine state and returns control to the guest.

Why did early x86 hardware make building a correct hypervisor difficult, and how did hardware virtualization extensions (VT-x/AMD-V) solve it?

<showhide>
The original x86 architecture had **non-virtualizable instructions**: some privileged instructions behaved differently when executed in user mode instead of silently trapping. A guest OS executing these instructions would get the "wrong" behavior without the hypervisor being notified. Early hypervisors (e.g., VMware's early products) used **binary translation**: scanning guest code and replacing non-virtualizable instructions with calls into the hypervisor. VT-x/AMD-V introduced a dedicated VMX non-root mode for guests where all sensitive instructions trap to the hypervisor, eliminating the need for binary translation.
</showhide>
