# Transport Layer and TCP

The transport layer provides end-to-end communication between applications. **TCP (Transmission Control Protocol)** provides reliable, ordered, byte-stream delivery over the unreliable IP layer. **UDP** provides unreliable datagrams for applications that need low latency and can tolerate loss (e.g., video streaming, DNS).

TCP achieves reliability through **sequence numbers** (detect reordering and duplication), **acknowledgments** (confirm receipt), and **retransmission** (resend lost segments). It achieves flow control via a **receive window** (the receiver tells the sender how much buffer space is available) and congestion control via algorithms like **AIMD (Additive Increase Multiplicative Decrease)**.

**Congestion control** prevents senders from overwhelming the network. TCP infers congestion from packet loss (timeout or three duplicate ACKs) and responds by reducing its sending rate. The congestion window grows additively during normal operation (1 MSS per RTT) and halves on congestion detection. This cooperative behavior, when used by all senders, leads to fair allocation of bandwidth.

## TCP Reliability Mechanisms

TCP's sliding window protocol allows multiple unacknowledged segments to be "in flight" simultaneously (up to the congestion window size). The sender maintains a window of sent but unacknowledged bytes. When an ACK arrives, the window slides forward. This keeps the pipe full: at any RTT, `window_size` bytes are in transit.

Why does TCP use a three-way handshake (SYN, SYN-ACK, ACK) rather than a two-way handshake to establish a connection?

<showhide>
A two-way handshake is vulnerable to delayed duplicates: an old SYN from a previous connection could arrive long after the original connection closed and cause the server to open a new connection with the wrong parameters. The three-way handshake lets both sides confirm that the other's chosen initial sequence number was received. The third ACK (from client) confirms to the server that the client received the SYN-ACK, preventing a server from getting stuck in LISTEN for old SYN packets.

## Congestion Control and AIMD

AIMD (Additive Increase, Multiplicative Decrease): increase cwnd by 1 MSS per RTT during normal operation; halve cwnd on congestion. This leads to a sawtooth throughput pattern. Under AIMD, multiple TCP flows sharing a bottleneck link converge to equal shares of the bandwidth—a fairness property.

Why does TCP use timeout-based retransmission in addition to fast retransmit (triple duplicate ACK), and what different scenarios does each address?

<showhide>
**Fast retransmit** (triple duplicate ACK) detects a single lost packet in a stream where other packets continue to arrive. Three duplicate ACKs indicate that the receiver got later packets but is missing one specific segment. **Timeout** handles the case where the network is severely congested and no packets (not even ACKs) are getting through—the receiver has no out-of-order packets to acknowledge. A timeout triggers slow start (cwnd=1), which is more aggressive than fast recovery because severe congestion requires a larger reduction in sending rate.
