# Caching and Cache-Efficient Algorithms

The memory hierarchy is central to performance. Modern CPUs have multiple levels of cache: L1 (≈32 KB, ~4 cycles), L2 (≈256 KB, ~12 cycles), L3 (≈8 MB, ~40 cycles), and DRAM (≈16+ GB, ~200 cycles). Algorithms must be designed to minimize cache misses, especially in the L1/L2 levels where the performance gap is largest.

The **ideal-cache model** is an analytic tool: it assumes a two-level memory hierarchy with a cache of $M$ words divided into cache lines of $B$ words. The cost of a computation is measured in cache misses. An algorithm is **cache-efficient** if it achieves close to the minimum possible number of cache misses.

## Cache Miss Analysis

Three types of cache misses (the "3 Cs"):
1. **Compulsory (cold) misses**: First access to a cache line — unavoidable.
2. **Capacity misses**: Working set doesn't fit in cache.
3. **Conflict misses**: Two addresses map to the same cache set (direct-mapped or set-associative).

For a matrix $A$ of $n \times n$ elements stored row-major, accessing $A[i][j]$ and $A[i][j+1]$ uses the same cache line, but $A[i][j]$ and $A[i+1][j]$ may be on different cache lines (stride $n$).

Cache miss count for matrix multiply (naive `ijk`):
- Matrix $A$: $\Theta(n^3/B)$ misses (accessed row-wise, OK)
- Matrix $B$: $\Theta(n^3)$ misses (accessed column-wise, one miss per element)
- Total: $\Theta(n^3)$ misses

What are "compulsory misses" and can they be avoided?

<showhide>
Compulsory (cold) misses occur on the first access to a cache line — regardless of cache size or replacement policy. They are unavoidable since data must be fetched from memory at least once. The only way to reduce them is to access fewer distinct cache lines (i.e., improve spatial locality so each cache line is used fully) or use prefetching to overlap the miss latency with computation.

## Blocked (Tiled) Matrix Multiply

Blocking (tiling) improves cache performance by processing the matrices in small blocks that fit in cache:

```c
// Blocked matrix multiply with block size B_size
for (int i = 0; i < n; i += B_size)
  for (int j = 0; j < n; j += B_size)
    for (int k = 0; k < n; k += B_size)
      // multiply B_size × B_size submatrices
      for (int ii = i; ii < i + B_size; ii++)
        for (int jj = j; jj < j + B_size; jj++)
          for (int kk = k; kk < k + B_size; kk++)
            C[ii][jj] += A[ii][kk] * B[kk][jj];
```

Choose block size $b$ so that three $b \times b$ blocks fit in cache: $3b^2 \leq M$, giving $b = \Theta(\sqrt{M/3})$. This gives $\Theta(n^3 / (B \sqrt{M}))$ cache misses, vs. $\Theta(n^3)$ for the naive approach — a factor of $B\sqrt{M}$ improvement.

What is "spatial locality" and "temporal locality" and how do they affect cache performance?

<showhide>
**Spatial locality**: accessing memory addresses near recently accessed addresses. Exploiting it means accessing arrays sequentially (stride-1), so the entire cache line is used after being fetched. **Temporal locality**: reusing a recently accessed memory location before it is evicted from cache. Exploiting it means working on a small dataset repeatedly (e.g., blocked algorithms keep a tile in cache and reuse it multiple times). Cache-efficient algorithms maximize both.
</showhide>
