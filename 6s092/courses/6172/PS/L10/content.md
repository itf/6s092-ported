# Measurement and Timing

Reliable performance measurement requires careful methodology. Clocks on modern systems have multiple sources of noise: OS scheduling jitter, memory hierarchy effects (cold vs. warm caches), CPU frequency scaling (turbo boost, power throttling), and measurement overhead itself. A single timing measurement is almost never sufficient; you must take multiple samples and understand the distribution.

The standard approach in 6.172 is to use `clock_gettime(CLOCK_MONOTONIC, &t)` for wall-clock time, or hardware performance counters (via tools like `perf`, `PAPI`, or `vtune`) for more detailed measurements.

## Timing Methodology

Key principles:
1. **Warm up the cache**: Run the computation once before timing to load data into cache.
2. **Repeat and take minimum or median**: The minimum time is often the most informative (least noise from OS interruptions).
3. **Disable CPU frequency scaling**: Set the governor to "performance" to avoid turbo boost variation.
4. **Use stable inputs**: Random inputs can cause branch mispredictions; control for this.
5. **Avoid measuring overhead**: Subtract the timing overhead if it is significant relative to what you're measuring.

```c
#include <time.h>
struct timespec start, end;
clock_gettime(CLOCK_MONOTONIC, &start);
// ... code to time ...
clock_gettime(CLOCK_MONOTONIC, &end);
double elapsed = (end.tv_sec - start.tv_sec) +
                 (end.tv_nsec - start.tv_nsec) * 1e-9;
```

Why is the *minimum* measured time often more meaningful than the *average* when benchmarking a code snippet?

<showhide>
The minimum time represents the case where the OS did not interrupt the process, the data was in cache, and the CPU ran at full speed without thermal throttling. The average includes random "noise" from OS scheduling, cache misses on cold runs, and other interference that inflates times. The minimum time is a better estimate of the true performance of the code. (The median can also be useful for robustness against outliers.)

## Tools: perf, valgrind, and Profilers

**`perf`** is a Linux tool for hardware performance counters. It can count cache misses, branch mispredictions, instructions retired, cycles, and more:
```bash
perf stat ./my_program        # summary statistics
perf record ./my_program      # record profile
perf report                   # interactive profile viewer
```

**Gprof** and **callgrind** (Valgrind) provide call-graph profiling. **`perf annotate`** shows which source lines consume the most cycles.

**Amdahl's Law** relates to profiling: if you speed up a fraction $f$ of the program by factor $S$, the overall speedup is:
$$\text{Speedup} = \frac{1}{(1-f) + f/S}$$

<checkyourself>
What is "cache warming" and why must you do it before timing a memory-intensive benchmark?

<showhide>
Cache warming means running the computation once (without timing) before the timed run, so that the data is loaded from main memory into the CPU caches. If you measure the first run, you're measuring cold-cache performance (including DRAM latency). The warm-cache run is usually much faster and more representative of steady-state performance (e.g., when a hot loop runs many times). Both numbers can be useful depending on your use case.
</showhide>
