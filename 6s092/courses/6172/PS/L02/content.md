# Bentley Rules for Optimizing Work

The *work* of a program on a given input is the total number of operations executed. Reducing work is a powerful strategy for improving performance, even though it doesn't automatically translate to faster execution (due to instruction-level parallelism, caching, and vectorization). Jon Bentley organized code optimization techniques into four categories: **Data structures**, **Loops**, **Logic**, and **Procedures**.

Reducing work often involves eliminating redundant computation, choosing better algorithms, or restructuring data to avoid wasted effort. While algorithmic improvements (e.g., replacing $\Theta(n^2)$ sort with $\Theta(n \log n)$ sort) can give asymptotic wins, Bentley rules target constant-factor improvements that compound to yield large practical speedups.

## Data Structure Rules

Key data structure optimizations include **packing and encoding** (storing multiple values in a single machine word), **augmenting** data structures to store precomputed results, and **caching** frequently computed values. For example, if you repeatedly compute `strlen(s)` inside a loop, cache the result in a variable before the loop.

```c
// Bad: recomputes strlen on every iteration
for (int i = 0; i < strlen(s); i++) { ... }

// Good: compute once
int n = strlen(s);
for (int i = 0; i < n; i++) { ... }
```

Example: precomputing a table of factorials or binomial coefficients for a combinatorics problem, rather than recomputing them during each query. ## Loop Rules

Loop optimizations include **hoisting** (moving loop-invariant computations out of the loop body), **sentinels** (using a special value to eliminate a boundary check), **loop unrolling** (reducing loop-control overhead and enabling more ILP), **loop fusion** (combining multiple loops over the same range), and **eliminating wasted iterations**.

Loop unrolling example:

```c
// Original
for (int i = 0; i < n; i++) sum += a[i];

// Unrolled by 4
for (int i = 0; i < n-3; i += 4)
    sum += a[i] + a[i+1] + a[i+2] + a[i+3];
for (int i = n - (n%4); i < n; i++) sum += a[i];
```

## Logic and Procedure Rules

**Logic rules** include exploiting algebraic identities (e.g., replacing `x * 2` with `x << 1`), short-circuit evaluation, and ordering tests by decreasing probability of success. **Procedure rules** include inlining small functions to eliminate call overhead, tail-recursion elimination, and coarsening recursion (handling small base cases iteratively).

What is the difference between "reducing work" and "reducing running time"? Why can reducing work fail to reduce running time?

<showhide>
Work is the total number of operations; running time depends on the critical path through the hardware. Reducing work may not reduce running time if: (1) the bottleneck is memory latency rather than compute, (2) the original code had good ILP and the optimized version serializes it, or (3) the "saved" operations were handled in parallel by superscalar/SIMD execution. Nonetheless, reducing work is a good heuristic because it often correlates with faster execution.
</showhide>
