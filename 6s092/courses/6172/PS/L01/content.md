# Introduction & Matrix Multiplication

Performance engineering is the practice of making programs run faster by carefully exploiting computer hardware. While correctness, clarity, and maintainability are often more important properties, performance is crucial in practice: many modern applications are bottlenecked by speed rather than developer time.

The classic example is matrix multiplication. The naive triple-loop algorithm for multiplying two $n \times n$ matrices has work $\Theta(n^3)$ and exhibits poor cache behavior. By reordering loops to improve spatial locality, or using blocked (tiled) variants, one can achieve dramatically better practical performance without changing the asymptotic complexity.

The key observation in this course is that writing asymptotically optimal algorithms is necessary but not sufficient—you must also understand the hardware to achieve real-world performance. A program with fewer operations but poor cache behavior can be many times slower than one with more operations but good locality.

## Why Performance Engineering?

Modern processors have deep memory hierarchies: registers, L1/L2/L3 caches, DRAM, and disk. The ratio of speeds between adjacent levels is often 10x or more. A program that fits its working set in L1 cache can run 100x faster than one that repeatedly goes to DRAM. Similarly, modern CPUs execute multiple operations per cycle using pipelining, superscalar execution, SIMD, and out-of-order execution—understanding these features is essential for squeezing out performance.

The naive matrix multiply in C:

```c
for (int i = 0; i < n; i++)
  for (int j = 0; j < n; j++)
    for (int k = 0; k < n; k++)
      C[i][j] += A[i][k] * B[k][j];
```

This `ijk` loop order accesses `B` column-wise (stride $n$), causing cache misses. Swapping the inner two loops to `ikj` order makes `B` access row-wise, dramatically improving cache performance.

<checkyourself>
Why does the `ikj` loop order for matrix multiply typically outperform `ijk` on modern hardware even though both do the same number of floating-point operations?

<showhide>
In the `ijk` order, the inner loop over `k` accesses `B[k][j]` with stride $n$ (column-wise), generating a cache miss on nearly every access. In the `ikj` order, the inner loop over `j` accesses `B[i][j+1]` sequentially (row-wise), benefiting from spatial locality and hardware prefetching. `A[i][k]` is also scalar-promoted from the middle loop, so all three arrays are accessed efficiently.

## Performance vs. Other Properties

Performance is one of many software properties. Compatibility, correctness, modularity, and maintainability are all typically more important. Performance engineering should only be applied after the program is correct. The famous quote by Knuth: "Premature optimization is the root of all evil." However, when performance does matter—and it often does at scale—knowing how to systematically improve it is invaluable.

<checkyourself>
What is "loop reordering" and why can it improve performance without changing the number of arithmetic operations?

<showhide>
Loop reordering changes the sequence in which array elements are accessed. Since modern hardware uses caches and prefetchers that work best with sequential (stride-1) access patterns, reordering loops to access arrays row-by-row (in row-major order) rather than column-by-column reduces cache misses significantly, improving performance without changing the total number of floating-point operations.
