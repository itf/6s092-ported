# Synchronization Without Locks

Lock-free (and wait-free) data structures allow concurrent access without holding a mutual exclusion lock. This avoids deadlock, priority inversion, and the performance overhead of lock contention. Lock-free algorithms rely on hardware atomic primitives, especially **Compare-And-Swap (CAS)**.

Understanding memory models is critical: on modern processors (including x86), hardware can reorder memory operations. The C11/C++11 memory model provides tools to specify the required ordering.

## Sequential Consistency

A **sequentially consistent** execution is one where all memory operations appear to execute in some global total order consistent with the program order of each thread. This is the simplest model to reason about: it's as if all threads share one global memory and alternate instructions atomically.

However, hardware and compilers may reorder operations for performance. On x86, reads are not reordered with reads, and writes are not reordered with writes, but stores can be reordered with subsequent loads. Using `mfence` (memory fence) or atomic operations with appropriate ordering prevents reordering.

<showhide>
Sequential consistency means all memory operations appear to execute in a single total order, consistent with the per-thread program order. It's easy to reason about but expensive to implement: processors would have to flush their store buffers and not speculate across memory operations. Modern hardware uses relaxed memory models (e.g., x86-TSO: stores may be delayed in a write buffer) because they are faster. Programs can force sequential consistency at specific points using memory fences or sequentially-consistent atomic operations.

## Compare-And-Swap (CAS)

CAS is the fundamental building block for lock-free algorithms:

```c
// Pseudocode for atomic CAS:
bool CAS(int *addr, int expected, int new_val) {
    if (*addr == expected) {
        *addr = new_val;
        return true;  // success
    }
    return false;  // failure: *addr was changed by another thread
}
```

On x86, this is the `cmpxchg` instruction with a `lock` prefix. The standard pattern for lock-free update:

```c
int old, new;
do {
    old = atomic_load(&x);
    new = compute(old);
} while (!CAS(&x, old, new));  // retry if someone else changed x
```

This is called an **optimistic CAS loop** or **read-modify-CAS** pattern. It has the **ABA problem**: if `x` changes from A to B and back to A between the load and CAS, the CAS succeeds but the data structure may be in an inconsistent state.

<checkyourself>
What is the difference between "lock-free" and "wait-free" algorithms?

<showhide>
**Lock-free**: At least one thread makes progress in a finite number of steps, even if some threads are delayed indefinitely. Individual threads may starve (retry many times), but the system as a whole progresses. The CAS retry loop is lock-free but not wait-free.
**Wait-free**: Every thread makes progress in a bounded number of steps, regardless of other threads' behavior. No thread can be starved. Wait-free algorithms are harder to design. Every wait-free algorithm is lock-free, but not vice versa.
</showhide>
</checkyourself>
