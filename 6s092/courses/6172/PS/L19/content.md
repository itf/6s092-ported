# Leiserchess Codewalk

Leiserchess is a two-player laser chess variant played on an 8×8 board. Each player has 7 Pawns and 1 King. Pieces have orientations (N/E/S/W) that determine how they deflect or are destroyed by lasers. The 6.172 final project has students optimize a Leiserchess AI engine, applying all the performance techniques from the course.

The engine uses **alpha-beta search** (a minimax tree search with pruning) to find the best move. Its performance is determined by how many positions it can evaluate per second, which depends on: the board representation, move generation, evaluation function, and the search algorithm (depth, pruning effectiveness, move ordering).

## Board Representation

The board is represented as a packed array of 64-bit words. Each cell contains piece type, color, and orientation. Compact representations enable fast copy, comparison, and manipulation. Common representations in chess engines:

- **Array of squares**: simple, cache-friendly for piece lookup by position.
- **Bitboards**: one 64-bit integer per (piece type, color) pair, with one bit per square. Fast for bulk operations (e.g., find all attacked squares using bit operations).

For Leiserchess, a key operation is laser tracing: given the King's laser, simulate the laser bouncing through the board until it hits something or goes off-board.

<showhide>
A bitboard represents the set of squares occupied by pieces of a given type/color as a 64-bit integer (one bit per square for an 8×8 board). Operations like "find all squares attacked by bishops" become bitwise operations on a few 64-bit words. Population count (popcount), bit scanning (finding the lowest/highest set bit), and set operations (intersection, union, complement) are all O(1) hardware operations, making many board queries extremely fast.

## Alpha-Beta Search

Alpha-beta pruning is an optimization of minimax search that prunes branches that cannot affect the final result. The key invariant: maintain a window $[\alpha, \beta]$; prune when the current score is outside this window.

```c
int alphabeta(Board *b, int depth, int alpha, int beta) {
    if (depth == 0) return eval(b);
    for each move m {
        make_move(b, m);
        int score = -alphabeta(b, depth-1, -beta, -alpha);
        undo_move(b, m);
        if (score >= beta) return beta;   // beta cutoff (prune)
        if (score > alpha) alpha = score;
    }
    return alpha;
}
```

**Move ordering** is critical: if the best move is searched first, alpha-beta prunes the most branches. With perfect move ordering, alpha-beta searches $O(b^{d/2})$ nodes instead of $O(b^d)$ (where $b$ is branching factor and $d$ is depth).

<checkyourself>
Why is move ordering important for alpha-beta pruning effectiveness?

<showhide>
Alpha-beta prunes a branch when it can prove that the opponent already has a better option elsewhere. This pruning is most effective when the best moves are examined first—because a good move found early establishes a tight $[\alpha, \beta]$ window, causing more subsequent branches to be pruned. If moves are examined in worst order (worst moves first), almost no pruning occurs and alpha-beta degenerates to full minimax. Good move ordering (e.g., using the transposition table, killer move heuristic, or history heuristic) is as important as the pruning algorithm itself.
</showhide>
</checkyourself>
