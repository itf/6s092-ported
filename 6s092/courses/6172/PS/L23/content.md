# High Performance in Dynamic Languages

Dynamic languages like Python, R, and MATLAB offer flexibility and productivity but are typically much slower than C/C++ for numerical computation. The traditional approach is the "two-language problem": write high-level code in Python, but performance-critical sections in C/Fortran/Cython. **Julia** was designed to solve this: achieve C-like performance with Python-like productivity, for scientific computing.

The key insight from this lecture (Steven Johnson, MIT Applied Mathematics): *vectorize* your code — rely on mature external libraries (BLAS, LAPACK, FFT) operating on large blocks of data. But someone has to write those libraries, and eventually you'll need to write performance-critical code yourself.

## The Two-Language Problem

Python is easy to write but slow for loops:

```python
# Slow: pure Python loop
def sum_squares(n):
    s = 0
    for i in range(n): s += i*i
    return s

# Fast: NumPy (calls BLAS/C under the hood)
import numpy as np
s = np.sum(np.arange(n)**2)
```

The NumPy version can be 100–1000× faster because it calls into optimized C/Fortran code. But it requires restructuring your computation as bulk array operations, which is not always natural.

## JIT Compilation and Type Inference

Julia's performance comes from **type inference** and **JIT compilation**. When a function is called for the first time, Julia infers the types of all variables and compiles a specialized version for those types to native machine code via LLVM. Subsequent calls with the same types use the cached compiled version.

```julia
# Julia: this compiles to efficient machine code
function sum_squares(n)
    s = 0
    for i in 1:n
        s += i*i
    end
    return s
end
```

Julia infers that `n` is `Int64`, `s` is `Int64`, and `i` is `Int64`, and generates efficient LLVM IR — comparable to C's `-O2` output. The key requirement: **type stability** — the return type of a function must be determinable from argument types alone.

<checkyourself>
Why can't a Python interpreter achieve C-like performance for a tight numerical loop, even with optimizations?

<showhide>
Python is dynamically typed: every variable is a Python object with a type tag and reference count. Every operation (e.g., `a + b`) must check the types of `a` and `b` at runtime, dispatch to the correct implementation, allocate a new Python object for the result, and update reference counts. A C loop does none of this: it operates directly on machine integers/floats in registers. This overhead is typically 50–100× per operation. CPython also holds the Global Interpreter Lock (GIL), preventing true parallel execution in threads.
</showhide>
</checkyourself>
