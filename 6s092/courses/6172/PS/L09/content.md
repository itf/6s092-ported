# What Compilers Can and Cannot Do

Modern compilers (Clang/LLVM, GCC) perform sophisticated optimizations, but they are fundamentally conservative: they can only apply a transformation if they can *prove* it is safe (preserves the observable behavior of the program). This lecture explores which optimizations compilers perform reliably and which they cannot, so that programmers know when to help the compiler.

The Clang/LLVM pipeline: source → Clang frontend (AST) → LLVM IR → optimization passes → backend code generation → assembly. The optimization passes operate on LLVM IR and include: constant folding, dead code elimination, inlining, loop-invariant code motion, alias analysis, auto-vectorization, and more.

## What Compilers Can Do

Compilers reliably perform:
- **Constant folding**: `3 * 4` → `12` at compile time.
- **Strength reduction**: `x * 2` → `x + x` or `x << 1`.
- **Common subexpression elimination**: reuse a computed value instead of recomputing.
- **Dead code elimination**: remove code whose result is never used.
- **Inlining**: replace a function call with the function body.
- **Loop-invariant code motion** (LICM): hoist computations out of loops when safe.

```c
// Compiler can optimize this:
int foo(int x) { return x * 16; }
// → shl rax, 4  (left shift by 4)
```

## What Compilers Cannot Do

Compilers cannot apply transformations that might change observable behavior. Key limitations:

1. **Aliasing**: If two pointers might alias, the compiler cannot reorder their accesses. Use `restrict` to inform the compiler.
2. **Floating-point reassociation**: `(a + b) + c ≠ a + (b + c)` in floating-point due to rounding. Compilers won't reorder unless you pass `-ffast-math` (which allows imprecise FP).
3. **Volatile and memory-mapped I/O**: `volatile` variables must be accessed exactly as written.
4. **Functions with side effects**: A call to an external function might modify global state; the compiler can't assume it doesn't.
5. **Integer overflow**: Signed integer overflow is **undefined behavior** in C; the compiler may assume it doesn't happen and optimize accordingly.

```c
// Compiler CANNOT vectorize this (potential aliasing):
void add(int *a, int *b, int *c, int n) {
    for (int i = 0; i < n; i++) c[i] = a[i] + b[i];
}

// With restrict, it CAN vectorize:
void add(int * restrict a, int * restrict b, int * restrict c, int n) {
    for (int i = 0; i < n; i++) c[i] = a[i] + b[i];
}
```

<checkyourself>
What does `-ffast-math` do, and what are the risks of using it?

<showhide>
`-ffast-math` (or `-Ofast`) allows the compiler to perform floating-point optimizations that may violate IEEE 754 semantics, including: reassociating operations, replacing divisions by multiplications by reciprocal, treating NaN/Inf as absent, and fusing multiply-add. The risk is that results may differ numerically from strict IEEE 754 computation, which can cause subtle correctness bugs in numerical code (e.g., convergence checks, physical simulations, financial calculations). Use with caution and only after verifying the numerical impact.
</showhide>
</checkyourself>
