# Parallel Storage Allocation

Memory allocation in parallel programs is significantly more challenging than in serial programs. The standard `malloc`/`free` uses a global lock, creating a bottleneck when many threads allocate simultaneously. Modern parallel allocators like **jemalloc** and **tcmalloc** use per-thread or per-processor caches to reduce contention.

A key challenge is that objects allocated by one thread may be freed by another ("producer-consumer" allocation patterns). The allocator must handle this without deadlock or excessive overhead.

## Heap Allocation Primitives

C heap allocation:
- `void *malloc(size_t s)` — allocate at least `s` bytes
- `void free(void *p)` — free a previously allocated block
- `void *realloc(void *p, size_t s)` — resize an allocation
- `void *calloc(size_t n, size_t s)` — allocate `n*s` bytes, zero-initialized

The operating system provides memory to the allocator in large chunks via `mmap` or `sbrk`. The allocator then sub-divides these chunks for individual requests.

<checkyourself>
Why is a global lock on `malloc` a bottleneck in parallel programs?

<showhide>
With a global lock, only one thread can allocate or free memory at a time. In a parallel program with many threads frequently allocating/freeing (e.g., task-parallel programs, web servers), threads spend time waiting for the lock. This serializes what should be parallel work and creates contention that grows with the number of threads. The lock also causes cache-line invalidations across processors, adding further overhead.
</showhide>
</checkyourself>

## Per-Thread Caches and Arenas

Modern allocators mitigate contention using **per-thread caches** (magazines or thread-local caches). Each thread has a fast-path local cache for common size classes. Allocations that hit the local cache require no locking:

```
Thread 0 → [local cache 0] ↘
Thread 1 → [local cache 1] → [shared arena] → OS
Thread 2 → [local cache 2] ↗
```

When the local cache is empty, the thread refills from a shared "arena" (with a finer-grained lock per arena). **jemalloc** uses multiple arenas assigned to threads to reduce contention. **tcmalloc** (Google) maintains a per-thread cache and a central free list.

**Cilk's parallel allocator** (cactus stack model) is designed for task-parallel programs where allocation/deallocation follows the call tree structure.

What is the "cactus stack" memory model used in Cilk, and how does it differ from the standard call stack?

<showhide>
In a standard serial program, the call stack is a single linear stack; each function call pushes a frame and each return pops it. In a Cilk parallel program, a spawned function may run in parallel with its parent, so their stack frames coexist simultaneously. The resulting structure is a tree (not a list) of stack frames — the "cactus stack." Each path from root to leaf in the tree is a valid call stack for one strand. The Cilk runtime must manage this tree structure rather than a simple array, which complicates memory allocation for stack frames.
</showhide>
</checkyourself>
