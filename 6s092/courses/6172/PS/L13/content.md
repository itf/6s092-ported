# The Cilk Runtime System

The Cilk runtime system is responsible for scheduling Cilk tasks onto available processor threads (workers). It implements the **randomized work-stealing** algorithm, which provably achieves near-optimal expected performance. Understanding the runtime helps you write Cilk programs that cooperate well with the scheduler.

Each Cilk worker thread maintains a **deque** (double-ended queue) of frames. Workers execute their own deque from the bottom (like a stack), preserving serial execution order for cache locality. When a worker runs out of work, it becomes a **thief** and steals from the **top** of a randomly chosen victim's deque.

## The Work-Stealing Mechanism

When a `cilk_spawn` is encountered:
1. The parent's continuation (the code after the spawn) is pushed onto the bottom of the deque.
2. The spawned function begins executing immediately.
3. If another worker steals the continuation, it runs in parallel with the spawned call.

When a `cilk_sync` is reached:
- If there are outstanding spawned children, the worker suspends and becomes available for stealing.
- Once all children complete, the worker (or any available worker) resumes after the sync.

This model ensures that serial execution (no stealing) follows the same order as a serial C program, preserving cache behavior in the common case.

In Cilk's work-stealing scheduler, which end of the deque does a worker use for its own work, and which end does a thief steal from? Why?

<showhide>
A worker pushes and pops from the **bottom** of its deque (LIFO — like a stack), which preserves the serial execution order and good cache behavior for the common case (no parallelism). A thief steals from the **top** of a victim's deque (FIFO), which tends to steal the oldest, largest tasks (those pushed earliest = deepest in the call tree). This gives thieves coarse-grained chunks of work to do, amortizing the steal overhead.

## Cilkscale: Parallel Performance Analyzer

**Cilkscale** (formerly Cilkview) is a tool that measures work $T_1$ and span $T_\infty$ of a Cilk program, allowing you to predict its performance on $P$ processors:

$$T_P \approx \frac{T_1}{P} + T_\infty$$

Cilkscale instruments your program and reports: work, span, parallelism, and a "performance chart" showing predicted speedup vs. number of processors.

The **overhead** of the Cilk runtime is low: on a serial execution (no stealing), the overhead per spawn is just a few instructions (to set up the deque entry). Stealing is rare in programs with high parallelism.

What is a "continuation" in the context of Cilk's runtime, and why must it be stored when a `cilk_spawn` occurs?

<showhide>
A continuation is the remainder of the computation after a spawn point — i.e., the code that will execute after the spawned function returns. When `cilk_spawn f()` is executed, the parent must save its continuation (current state: program counter, register values, local variables) so that execution can resume there later. In Cilk, the continuation is pushed onto the worker's deque and may be stolen by another worker to run in parallel. Without saving the continuation, there would be no way to resume the parent after the spawned function completes.
</showhide>
