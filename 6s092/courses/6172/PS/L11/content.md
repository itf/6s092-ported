# Storage Allocation

Efficient memory management is critical for performance. The C standard library provides `malloc`/`free` for heap allocation, but their general-purpose design means they can be slow (due to locking, fragmentation, and bookkeeping overhead). Understanding how allocators work enables you to design faster special-purpose allocators.

There are three main storage classes: **stack** (automatic variables, very fast LIFO allocation), **heap** (dynamic allocation via `malloc`/`free`), and **static** (global/static variables, allocated at program start).

## Stack Allocation

Stack allocation is extremely fast: it's just a pointer increment/decrement. The function call frame uses the stack. However, stack size is limited (typically 1–8 MB) and the lifetime must be LIFO.

```c
void foo() {
    int a[1000]; // stack-allocated: fast, freed automatically on return
    int *b = malloc(1000 * sizeof(int)); // heap: slower, must free manually
    // ...
    free(b);
}
```

Variable-length arrays (VLAs) in C99 allocate on the stack at runtime. They're convenient but can cause stack overflow for large sizes and make stack frame size unpredictable.

What are the advantages of stack allocation over heap allocation?

<showhide>
Stack allocation is: (1) O(1) time — just increment/decrement a stack pointer; (2) automatically freed when the function returns (no memory leaks); (3) excellent cache locality — stack frames for recently called functions are likely in L1 cache; (4) no fragmentation. The downsides are limited size and LIFO lifetime constraint.

## Heap Allocators

A heap allocator manages a large pool of memory, handling requests of varying sizes. Key design concerns:
- **Space efficiency**: minimize wasted space (fragmentation).
- **Time efficiency**: minimize time per allocation/free.
- **False sharing**: avoid placing objects from different threads in the same cache line.

**Free list** allocators maintain linked lists of free blocks per size class. `malloc` finds a free block of the right size; `free` returns the block to the free list. **Binned free lists** (used by most modern allocators like jemalloc, tcmalloc) have a separate free list for each size class (e.g., 8, 16, 32, 64, ... bytes).

**Fragmentation** types:
- **Internal**: allocated block is larger than requested (wasted space within the block).
- **External**: free blocks too small/scattered to satisfy a large request (wasted space between blocks).

What is "false sharing" in the context of memory allocation, and how can it hurt parallel performance?

<showhide>
False sharing occurs when two threads access different data that happens to reside on the same cache line (64 bytes on x86). Even though they're accessing different variables, whenever one thread writes, the cache line is invalidated in the other processor's cache, forcing a cache-coherence round trip. This can be as slow as a cache miss even though no true data sharing occurs. Mitigations: pad objects to cache line size, use per-thread allocators (like tcmalloc) to avoid placing different threads' objects on the same cache line.
</showhide>
</checkyourself>
