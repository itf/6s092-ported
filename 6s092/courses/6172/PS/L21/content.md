# Tuning a TSP Algorithm

This lecture, delivered by Jon Bentley (Bell Labs), demonstrates systematic algorithm engineering applied to the **Traveling Salesperson Problem (TSP)**. TSP is NP-hard in general, but for practical instances, a sequence of algorithmic improvements can yield enormous speedups. The lesson is that performance engineering is as much about *algorithmic* choices as hardware-level tuning.

The classic approach: generate all permutations of cities, compute tour length, keep the minimum. This has $\Theta(n!)$ complexity — impractical for $n > 12$. A sequence of improvements achieves practical performance for $n$ up to hundreds or thousands.

## Sequence of Improvements

Bentley's sequence of TSP optimizations:

1. **Brute force**: $\Theta(n!)$ — enumerate all permutations.
2. **Fix starting city**: reduce by factor $n$ → $\Theta((n-1)!)$.
3. **Branch-and-bound**: prune branches whose partial cost exceeds the best known tour.
4. **Dynamic programming (Held-Karp)**: $\Theta(n^2 2^n)$ — optimal for small $n$.
5. **Nearest-neighbor heuristic**: $O(n^2)$ greedy, gives ~25% suboptimal tours.
6. **2-opt local search**: iteratively remove crossing edges; fast in practice.
7. **Spatial data structures**: use k-d trees or neighbor lists to find nearby cities in $O(\log n)$.

Each step can yield 10–1000× speedups. The total improvement from step 1 to a practical heuristic can be $10^{20}$ or more.

Why is "branch-and-bound" effective for TSP even though TSP is NP-hard?

<showhide>
Branch-and-bound prunes the search tree: whenever a partial tour's cost already exceeds the best complete tour found so far, you can discard the entire subtree. With a good lower bound and a good initial solution (from a heuristic), most of the exponential search tree is pruned. In practice, branch-and-bound solves TSP instances with hundreds of cities quickly, even though worst-case complexity is still exponential.
</showhide>

## Principles of Algorithm Engineering

Bentley's principles for systematically improving an algorithm:

1. **Profile before optimizing**: find the actual bottleneck, not what you assume it is.
2. **Use asymptotically better algorithms**: a better algorithm beats any micro-optimization.
3. **Apply Bentley rules**: hoist loop invariants, precompute, use bit tricks.
4. **Use efficient data structures**: a hash table or k-d tree can replace $O(n^2)$ with $O(n \log n)$.
5. **Use spatial locality**: organize data to reduce cache misses.
6. **Exploit problem structure**: TSP tours don't cross themselves (2-opt removes crossing edges).

```c
// 2-opt improvement: remove two edges and reconnect if shorter
bool improved = true;
while (improved) {
    improved = false;
    for (int i = 0; i < n-1; i++)
        for (int j = i+2; j < n; j++)
            if (dist(tour[i], tour[j]) + dist(tour[i+1], tour[j+1])
                < dist(tour[i], tour[i+1]) + dist(tour[j], tour[j+1])) {
                reverse(tour, i+1, j);
                improved = true;
            }
}
```

What is the "nearest-neighbor heuristic" for TSP, and what is its typical solution quality compared to optimal?

<showhide>
The nearest-neighbor heuristic: start at any city, repeatedly go to the nearest unvisited city, then return to the start. It runs in $O(n^2)$ time (or $O(n \log n)$ with a k-d tree). The solution quality is typically 20–25% above optimal for random instances. It's not guaranteed to be within any constant factor of optimal in the worst case, but it's fast and gives a good starting point for local search (e.g., 2-opt improvement).
</showhide>
</checkyourself>
