# Cache-Oblivious Algorithms

**Cache-aware** algorithms (like blocked matrix multiply) require tuning the tile size $b$ to the specific cache size $M$ and cache line size $B$. **Cache-oblivious** algorithms achieve optimal cache complexity *without knowing* $M$ or $B$ — they use recursive divide-and-conquer, and the optimal blocking arises naturally for all levels of the memory hierarchy simultaneously.

The cache-oblivious paradigm, pioneered by Frigo, Leiserson, Prokop, and Ramachandran, is especially powerful on machines with multi-level caches, since a single algorithm can be optimal for L1, L2, L3, and DRAM without separate tuning for each level.

## Cache-Oblivious Matrix Multiply

Instead of fixed-size tiles, recursively divide:

```c
// Cache-oblivious matrix multiply: split the largest dimension
void mm(int n, double *A, double *B, double *C) {
    if (n == 1) { C[0] += A[0] * B[0]; return; }
    int h = n/2;
    // Split A by rows, B by columns:
    mm(h, A00, B00, C00);  mm(h, A01, B10, C00);
    mm(h, A00, B01, C01);  mm(h, A01, B11, C01);
    mm(h, A10, B00, C10);  mm(h, A11, B10, C10);
    mm(h, A10, B01, C11);  mm(h, A11, B11, C11);
}
```

When the subproblem fits in cache, all accesses become cache hits. The number of cache misses is $\Theta(n^3 / (B\sqrt{M}))$ — same as optimal cache-aware blocking, but without knowing $B$ or $M$.

What is the key insight that makes cache-oblivious algorithms work across all cache levels simultaneously?

<showhide>
When a recursive subproblem is small enough to fit in cache (at any level), all subsequent accesses to that subproblem are cache hits. The recursion naturally reaches the right size for each cache level — and since the recursion works for any $M$ and $B$, it is optimal for all levels of the hierarchy simultaneously. There is no need to know the cache parameters; the divide-and-conquer does the right thing automatically.

## Cache-Oblivious Stencil Computation

The heat diffusion stencil (2D): $u[t+1][i][j] = \frac{1}{5}(u[t][i][j] + u[t][i-1][j] + u[t][i+1][j] + u[t][i][j-1] + u[t][i][j+1])$

A naive implementation of $T$ time steps on an $N \times N$ grid:
- Reads: $5TN^2$ elements
- Cache misses: $\Theta(TN^2/B)$ if the grid fits in cache, but $\Theta(TN^2)$ if it doesn't.

The **cache-oblivious stencil** uses a space-time trapezoid decomposition (Frigo & Strumpen). For a 1D stencil over $n$ points for $t$ time steps, the trapezoid cuts achieve $\Theta(tn/B + t\sqrt{n})$ cache misses, vs. $\Theta(tn/B + tn/\sqrt{M})$ for the naive approach.

What is a "space-time trapezoid" in the context of cache-oblivious stencil computation?

<showhide>
A space-time trapezoid is a region of the space-time grid (spatial coordinates × time steps) with a trapezoidal shape. When computing a stencil over $n$ grid points for $t$ time steps, instead of computing all points at each time step (which thrashes cache for large $n$), you recursively split the trapezoid into two halves in either the space or time dimension. The base case is when the spatial extent is small enough to fit in cache. This achieves much better cache reuse than the naive time-step-by-time-step approach.
</showhide>
