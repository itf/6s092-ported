# Graph Optimization

Graphs are fundamental data structures used in social networks, road networks, web graphs, and scientific computing. Graph algorithms present unique performance challenges: irregular memory access patterns, poor cache behavior, and low arithmetic intensity. Optimizing graph algorithms requires careful attention to representation, memory layout, and parallelism.

A graph $G = (V, E)$ has $|V|$ vertices and $|E|$ edges. Common operations: BFS/DFS traversal, connected components, shortest paths, and PageRank.

## Graph Representations

**Adjacency matrix**: $|V| \times |V|$ matrix; $O(1)$ edge lookup but $O(V^2)$ space — impractical for large sparse graphs.

**Adjacency list**: for each vertex, store a list of its neighbors. Space $O(V + E)$. The standard implementation:

```c
// CSR (Compressed Sparse Row) representation
int offsets[n+1];  // offsets[v] = start of v's neighbor list
int edges[m];      // concatenated neighbor lists
// Neighbors of v: edges[offsets[v] .. offsets[v+1]-1]
```

**CSR (Compressed Sparse Row)** is cache-friendly: all neighbors of a vertex are contiguous in memory. Iterating over neighbors of all vertices accesses `edges` sequentially.

Adjacency matrix: $|V|^2 = 10^{12}$ entries — far too large (even 1 bit per entry = 125 GB).
CSR: $|V| + 1$ offset entries + $|E|$ edge entries = $\approx 10^6 + 10^7 \approx 10^7$ integers = ~80 MB (with 8-byte ints). For sparse graphs, CSR is dramatically more memory-efficient.

## Breadth-First Search (BFS)

BFS is a classic graph traversal. In a sequential setting, BFS runs in $O(V + E)$. In parallel, BFS is challenging because the frontier expands level by level (creating a synchronization barrier per level):

```c
// Sequential BFS
void bfs(int src, int *dist, int *offsets, int *edges, int n) {
    memset(dist, -1, n * sizeof(int));
    dist[src] = 0;
    int queue[n]; int head = 0, tail = 0;
    queue[tail++] = src;
    while (head < tail) {
        int v = queue[head++];
        for (int i = offsets[v]; i < offsets[v+1]; i++) {
            int u = edges[i];
            if (dist[u] == -1) { dist[u] = dist[v] + 1; queue[tail++] = u; }
        }
    }
}
```

**Parallel BFS** (Beamer's direction-optimizing BFS): switch between "top-down" (expand frontier outward) and "bottom-up" (each unvisited vertex checks if any of its neighbors is in the frontier). Bottom-up is faster when the frontier is large.

<checkyourself>
What is "direction-optimizing BFS" (Beamer et al.) and when does it help?

<showhide>
Standard "top-down" BFS expands each vertex in the frontier and adds its unvisited neighbors to the next frontier. When the frontier is large (near the middle of BFS on a social network), this is wasteful: most neighbors are already visited. "Bottom-up" BFS instead has each *unvisited* vertex check if any of its neighbors is in the current frontier—avoiding re-examining already-visited vertices. The direction-optimizing approach switches from top-down to bottom-up when the frontier is large (saving work) and back when the frontier shrinks. On power-law graphs, this gives 10–60× speedups over standard BFS.
</showhide>
</checkyourself>
