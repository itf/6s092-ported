# Speculative Parallelism & Leiserchess

**Speculative parallelism** executes work that *might* be needed before knowing for certain that it is needed. If the speculation was correct, work completes faster. If wrong, the speculative work is discarded. This is especially relevant for alpha-beta search, where pruning is data-dependent—you can speculatively search branches that might get pruned.

A key enabling technique is **short-circuit evaluation**: when a sum (or other aggregate) threshold is met early, you can abort the computation and return. This is a Bentley rule for loops.

## Speculative Alpha-Beta (Parallel Search)

The main challenge of parallelizing alpha-beta: the pruning in alpha-beta is *serial* by nature—you learn that a branch can be pruned only after searching the first child, which gives you a better $\alpha$. Parallelizing naively loses the pruning.

**Young Brothers Wait** (YBW) and **Lazy SMP** are techniques to add parallelism while preserving most of the pruning benefit:

```
YBW: search first child serially to get a good alpha bound,
     then search remaining children in parallel.
```

With Cilk, you can speculatively spawn all children and use an `abort` mechanism to cancel spawned tasks when a beta cutoff is detected.

<checkyourself>
Why is alpha-beta search inherently difficult to parallelize?

<showhide>
Alpha-beta pruning is sequential: to know which branches can be pruned, you first need a bound $\alpha$ from searching the best move. This bound only becomes tight after searching the first (hopefully best) child. If you search all children in parallel, you lose the pruning benefit of sequential search. The result is that parallel alpha-beta with naive parallelism often searches *more* nodes than serial alpha-beta, even though it uses more cores. Techniques like Young Brothers Wait try to preserve pruning while adding limited parallelism.
</showhide>

## Transposition Tables

A **transposition table** is a hash table that stores the results of previously evaluated positions. If the same board position is reached via different move sequences (transpositions), the stored result can be reused:

```c
typedef struct {
    uint64_t hash;  // Zobrist hash of position
    int depth;      // search depth when evaluated
    int score;      // evaluation result
    int flag;       // EXACT, LOWER_BOUND, or UPPER_BOUND
} TTEntry;

TTEntry tt[TT_SIZE];  // fixed-size, collision-tolerant hash table
```

Zobrist hashing: assign a random 64-bit number to each (piece, square) combination; XOR them together for the whole board. Incremental updates: when a piece moves, XOR out the old position and XOR in the new one.

What is a transposition table and why is it important for game tree search performance?

<showhide>
A transposition table stores previously computed search results indexed by board position hash. Many board positions in a game tree can be reached via different sequences of moves (transpositions). Without a TT, the engine redundantly re-searches positions it has already evaluated. With a TT, it looks up the cached result and avoids re-searching, potentially saving enormous amounts of work. The speedup from a good TT can be 10× or more. TT entries also provide move suggestions that improve move ordering, which further improves pruning.
</showhide>
