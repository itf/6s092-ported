# C to Assembly

Reading the assembly generated by a C compiler is an essential performance-engineering skill. It reveals what the compiler actually did—and what it failed to do. Bugs can appear only at `-O3` because the compiler made an incorrect but valid-seeming transformation; examining assembly uncovers these. The Clang/LLVM pipeline compiles C → preprocessed source → LLVM IR → selection DAG → machine code.

The compiler applies a sequence of optimization passes on the LLVM IR before generating assembly. Understanding which passes the compiler can and cannot apply helps you write C code that enables better optimization.

## Reading Clang Output

Compile with `clang -O2 -S foo.c -o foo.s` to get annotated assembly. Use `objdump -d` to disassemble a binary. The flags `-O0`, `-O1`, `-O2`, `-O3` control optimization level; `-O2` is the standard production setting.

A simple example—the Fibonacci function:

```c
int fib(int n) {
    if (n < 2) return n;
    return fib(n-1) + fib(n-2);
}
```

At `-O0`, the compiler generates a literal recursive implementation. At `-O2`, it may still be recursive but with less stack manipulation. The compiler will not automatically transform this to iterative because it cannot prove it safe.

Why is examining the compiler-generated assembly useful for finding performance bugs, even if you don't plan to hand-write assembly?

<showhide>
The assembly shows exactly what the CPU executes. You can verify whether the compiler vectorized a loop, whether it unrolled it, whether it hoisted a loop-invariant computation, and whether it eliminated a redundant load. If you expected auto-vectorization but see scalar code, the assembly tells you why (e.g., potential aliasing). Bugs visible only at `-O3` often manifest as incorrect register reuse or incorrect alias assumptions visible in the assembly.
</showhide>

## Type Casts and Implicit Conversions

C's implicit type promotion rules can generate unexpected instructions. For example, mixing `int` and `unsigned int` in comparisons causes signed/unsigned conversions. Examining assembly helps catch cases where the compiler inserts zero-extension (`movzx`) or sign-extension (`movsx`) instructions unexpectedly.

```c
// This loop may not vectorize due to signed/unsigned mismatch:
for (int i = 0; i < n; i++) a[i] = b[i] + c[i];
// Using size_t can help the compiler generate better code for large n
for (size_t i = 0; i < (size_t)n; i++) a[i] = b[i] + c[i];
```

What is the `restrict` keyword in C99, and how does it help the compiler?

<showhide>
`restrict` is a hint to the compiler that the pointed-to object is accessed only through that pointer (no aliasing). For example, `void add(float * restrict a, const float * restrict b, int n)` tells the compiler that `a` and `b` do not overlap. With this guarantee, the compiler can safely vectorize the loop, reorder loads and stores, and generate significantly faster code.
</showhide>
</checkyourself>
