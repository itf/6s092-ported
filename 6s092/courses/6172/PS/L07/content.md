# Races and Parallelism

A **determinacy race** (or race condition) occurs when two logically parallel instructions access the same memory location, and at least one of them writes. Races are a primary source of bugs in parallel programs because they produce nondeterministic results—the program may give different answers on different runs or even on the same input.

Cilk provides the **Cilksan** race detector to find races at runtime. When a race is detected, Cilksan reports the two instructions involved and their logical parallel relationship. Eliminating races is necessary for correctness; races can also hurt performance (e.g., through cache coherence traffic).

## Types of Races

Given two operations $A$ and $B$ that are logically parallel (neither happens-before the other):
- **Read-Write race**: $A$ reads and $B$ writes the same location.
- **Write-Write race**: Both $A$ and $B$ write the same location.

Read-Read pairs are **not** races because concurrent reads are always safe.

```c
// Classic race: parallel increments
int x = 0;
cilk_spawn [&]{ x++; }();   // x++ is read-modify-write: not atomic!
x++;
cilk_sync;
// x may be 1 or 2 -- a race!
```

The `x++` operation expands to three machine instructions: load, increment, store. If two processors interleave these, the result is unpredictable.

A read-write race occurs when one instruction reads and another (logically parallel) instruction writes the same location. The read may see either the old or new value depending on scheduling, making the result nondeterministic. Write-write races are also problematic: only one write "wins," and which one depends on scheduling.

## Cilksan Race Detector

Cilksan instruments your program to detect races at runtime. Compile with `-fsanitize=cilk` (or the equivalent Cilk flags) and run your program. If a race is detected, Cilksan prints the two racing accesses and their location in the source.

**Important:** Cilksan only detects races on the particular execution it observes. It may miss races that don't occur during that run. The absence of a Cilksan report does not guarantee the program is race-free.

Reducers are a Cilk mechanism for eliminating races on accumulator variables. A `CILK_C_REDUCER_OPADD` reducer allows multiple parallel strands to independently update a local view and then combine the results during `cilk_sync`, avoiding the race:

```c
CILK_C_REDUCER_OPADD(sum, int, 0);
cilk_for (int i = 0; i < n; i++)
    REDUCER_VIEW(sum) += a[i];
// After cilk_sync (implicit at end of cilk_for), sum has the correct total
```

What is a Cilk reducer, and how does it eliminate a race condition on an accumulator variable?

<showhide>
A Cilk reducer is a special variable that gives each parallel strand its own private "view." When strands are created (spawned), each gets a fresh identity value (e.g., 0 for addition). Each strand independently updates its view. When strands join (at `cilk_sync`), the views are combined using the reducer's operation (e.g., addition). The final result is the same as serial execution but without any race, because no strand ever reads or writes another strand's view.
</showhide>
</checkyourself>
