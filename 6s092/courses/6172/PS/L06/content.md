# Multicore Programming

Moore's Law has continued to increase transistor density, but clock frequency scaling ended around 2004 due to power and heat limitations. Since then, chip vendors have responded by adding more cores per chip. Exploiting multiple cores requires writing parallel programs.

Cilk is a task-parallel extension to C/C++ developed at MIT. It adds three keywords: `cilk_spawn` (to indicate a child function may run in parallel with its parent), `cilk_sync` (to wait for all spawned children), and `cilk_for` (a parallel loop). The Cilk runtime schedules tasks onto available processor cores using work-stealing.

## Cilk Basics

```c
// Serial Fibonacci
int fib(int n) {
    if (n < 2) return n;
    int x = fib(n-1);
    int y = fib(n-2);
    return x + y;
}

// Parallel Fibonacci using Cilk
int fib(int n) {
    if (n < 2) return n;
    int x = cilk_spawn fib(n-1);
    int y = fib(n-2);
    cilk_sync;
    return x + y;
}
```

`cilk_spawn` grants *permission* for the spawned call to execute in parallel; it does not guarantee parallel execution. The runtime decides whether to actually run it in parallel based on available workers.

What is the difference between `cilk_spawn` and a regular function call?

<showhide>
A regular function call is synchronous: the caller suspends and the callee runs to completion before the caller continues. `cilk_spawn` is asynchronous: the spawned function *may* run in parallel with the code following the spawn in the parent. The parent can continue executing until it hits `cilk_sync`, which waits for all outstanding spawned children to return. The runtime may or may not actually run the spawn in parallel depending on available processors.

## Work-Stealing Scheduler

The Cilk runtime uses a **work-stealing** scheduler. Each processor has a deque (double-ended queue) of tasks. Processors work from the bottom of their own deque (LIFO, like a stack—good for cache locality). When a processor runs out of work, it randomly picks another processor and *steals* from the top of its deque (FIFO).

This strategy achieves near-optimal performance: the expected running time on $P$ processors is $T_P \leq T_1/P + O(T_\infty)$, where $T_1$ is the work (serial execution time) and $T_\infty$ is the span (critical path length).

<checkyourself>
What is the "parallel slackness" rule of thumb and how should it inform how you design parallel algorithms?

<showhide>
The parallel slackness rule suggests that maximum parallelism ($T_1/T_\infty$) should be at least 10× the number of processors you plan to use, so the scheduler has enough independent work to keep all processors busy despite scheduling overhead and variance. If parallelism is too low, some processors will sit idle waiting for the critical path. When designing parallel algorithms, aim to minimize span while keeping work at most a small constant factor above the serial work.
</showhide>
