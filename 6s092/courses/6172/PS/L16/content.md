# Nondeterministic Parallel Programming

A program is **deterministic** if every memory location is updated with the same sequence of values in every execution on the same input. Determinism is a valuable property: bugs are reproducible, testing is simpler, and reasoning about correctness is easier. The **Golden Rule of Parallel Programming**: *Never write nondeterministic parallel programs* — unless you have a compelling reason.

However, some parallel programs are intentionally nondeterministic — for example, programs using **mutexes (locks)** to coordinate shared-state updates. When using locks, the order in which threads acquire locks is not specified, leading to different execution orders across runs.

## Mutual Exclusion and Locks

A **mutex** (mutual exclusion lock) ensures that only one thread executes a critical section at a time:

```c
pthread_mutex_t lock;
pthread_mutex_lock(&lock);
// critical section: only one thread here at a time
shared_counter++;
pthread_mutex_unlock(&lock);
```

Using locks correctly requires avoiding **deadlock** (circular waiting on locks), **livelock** (threads keep changing state but never make progress), and **priority inversion** (high-priority thread blocked waiting for a lock held by a low-priority thread).

## Atomics and Memory Models

**Atomic operations** (like `__sync_fetch_and_add`, C11's `_Atomic`, or `std::atomic<T>`) provide hardware-supported operations that are indivisible — no other thread can see an intermediate state. They are faster than locks for simple operations:

```c
// Atomic increment (thread-safe, no lock needed)
_Atomic int counter = 0;
// in parallel:
atomic_fetch_add(&counter, 1);
```

However, atomic operations have ordering implications. The **memory model** specifies when writes by one thread become visible to others. `memory_order_seq_cst` (sequentially consistent) is the strongest and easiest to reason about but may be slowest; `memory_order_relaxed` is fastest but gives the weakest guarantees.

What is a "data race" (in the C11/C++11 sense), and how does it differ from a "determinacy race"?

<showhide>
A **C11 data race** occurs when two threads access the same non-atomic variable concurrently and at least one write is involved, with no synchronization (lock, atomic, etc.) between them. It is **undefined behavior** in C11 — the compiler can assume it doesn't happen and optimize accordingly. A **determinacy race** (Cilk sense) is broader: it occurs when two logically parallel strands access the same memory location and at least one writes, even if the result is deterministic (same value written). Every C11 data race is a determinacy race, but not vice versa (e.g., Cilk reducers eliminate determinacy races safely without a C11 data race).
</showhide>
</checkyourself>
