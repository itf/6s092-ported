# Domain-Specific Languages and Autotuning

Domain-specific languages (DSLs) capture programmer intent at a higher level of abstraction than C/C++, enabling compilers to apply domain-specific optimizations that a general-purpose compiler cannot. DSLs provide clarity, portability, and—crucially—the ability to encode expert knowledge about a domain into the compiler.

**Autotuning** is the systematic search over a space of implementation parameters (tile sizes, unroll factors, loop orderings, SIMD widths, etc.) to find the combination that runs fastest on a specific machine. Rather than manually tuning, autotuning automates the empirical search.

## Why DSLs?

A general-purpose compiler like Clang must conservatively assume any semantics the programmer might intend. A DSL for image processing *knows* that pixels are independent, arrays don't alias, and reductions are associative — enabling transformations that are unsafe in general C.

Examples of performance-oriented DSLs:
- **Halide**: image processing pipelines. Separates the *algorithm* from the *schedule* (when/where each value is computed, how loops are ordered and tiled).
- **FFTW**: FFT library that autotunes to the machine's memory hierarchy.
- **ATLAS**: autotuned BLAS linear algebra.
- **OpenCL/CUDA**: GPU computing DSLs.

```python
# Halide example: separable blur
blur_x = blur_x(x, y) = (input(x-1,y) + input(x,y) + input(x+1,y)) / 3
blur_y = blur_y(x, y) = (blur_x(x, y-1) + blur_x(x,y) + blur_x(x, y+1)) / 3
# Schedule: tile, vectorize, parallelize
blur_y.tile(x, y, xo, yo, xi, yi, 256, 32)
       .vectorize(xi, 8).parallel(yo)
```

What is the "algorithm vs. schedule" separation in Halide, and why is it useful?

<showhide>
In Halide, the **algorithm** specifies *what* is computed (the mathematical definition of each stage, like a blur or edge detection), while the **schedule** specifies *how* and *when* it is computed (loop order, tiling, vectorization, parallelization, fusion). Separating these lets you write the algorithm once in a clean, readable form, then explore many schedules to optimize performance without changing the correctness of the algorithm. The same Halide program can run efficiently on different machines by changing only the schedule.

## Autotuning

Autotuning explores the *parameter space* of an implementation:
- **Tile sizes**: affects cache hit rates.
- **Unroll factors**: affects loop overhead and ILP.
- **Vectorization width**: affects SIMD utilization.
- **Thread counts and work distribution**.

The parameter space can be enormous (exponential), so autotuning uses:
- **Exhaustive search**: for small spaces only.
- **Genetic algorithms / Bayesian optimization**: for large spaces.
- **Machine-learning models**: learn a performance model from a few samples.

FFTW (the "Fastest Fourier Transform in the West") uses autotuning to select among many internally coded FFT algorithms at install time. It achieves near-peak performance on most machines.

What is the main challenge with autotuning when the parameter space is very large?

<showhide>
When the parameter space is large (e.g., tile sizes for 3 nested loops × unroll factors × thread counts = thousands or millions of combinations), exhaustive search becomes infeasible. Each configuration may take seconds to benchmark, making exhaustive search take days. Solutions include: using heuristics to prune the space, surrogate models (predict performance from features without running all combinations), Bayesian optimization (model the performance surface and pick configurations likely to be good), or genetic algorithms.
</showhide>
