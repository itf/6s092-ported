# Cryptography: Hash Functions

**Cryptographic hash functions** are deterministic functions $H: \{0,1\}^* \to \{0,1\}^n$ with three security properties: (1) **preimage resistance** (one-wayness): given $y$, hard to find $x$ with $H(x) = y$; (2) **second preimage resistance**: given $x$, hard to find $x' \neq x$ with $H(x) = H(x')$; (3) **collision resistance**: hard to find any pair $x \neq x'$ with $H(x) = H(x')$.

These properties are ordered by strength: collision resistance $\Rightarrow$ second preimage resistance, and (roughly) second preimage resistance $\Rightarrow$ preimage resistance. Collision resistance does not imply preimage resistance in general.

The **birthday paradox** determines the security level: after approximately $2^{n/2}$ random queries, a collision is found with constant probability. Thus a 256-bit hash (like SHA-256) provides 128-bit collision resistance—requiring $2^{128}$ work to find a collision.

## Birthday Paradox and Collision Finding

The birthday paradox: among $k$ random values in $\{0,1\}^n$, the probability of a collision is $\geq 1/2$ when $k \approx 2^{n/2}$. Proof: the probability that all $k$ values are distinct is $\prod_{i=0}^{k-1}(1 - i/2^n) \approx e^{-k(k-1)/(2 \cdot 2^n)}$. Setting this $\leq 1/2$ gives $k \approx \sqrt{2 \ln 2} \cdot 2^{n/2}$.

For MD5 (128-bit output), collisions require $\approx 2^{64}$ operations; for SHA-1 (160-bit), $\approx 2^{80}$; for SHA-256, $\approx 2^{128}$ (currently infeasible).

Why does collision resistance not imply preimage resistance?

<showhide>
Consider $H'(x) = H(x)$ for $|x| > n$ and $H'(x) = x$ for $|x| \leq n$ (where $H$ is collision-resistant). Then $H'$ is collision-resistant (hard to find $H'(x) = H'(x')$ for $x \neq x'$), but it is NOT preimage-resistant: given $y = H'(y)$ for short $y$ (with $|y| \leq n$), we have the preimage $x = y$ trivially. So collision resistance is about finding any pair, while preimage resistance is about inverting, and neither implies the other in general.

## Merkle-Damgård Construction

Most practical hash functions (MD5, SHA-1, SHA-2) use the **Merkle-Damgård construction**: split the message into blocks, apply a **compression function** $f: \{0,1\}^{n+b} \to \{0,1\}^n$ iteratively (where $b$ is the block size), with an initial value $IV$ and Merkle-Damgård strengthening (padding the message length).

**Security**: If $f$ is a collision-resistant compression function, then the Merkle-Damgård hash is collision-resistant. This reduces the security of the full hash to the security of the compression function.

**Length extension attacks**: Merkle-Damgård is vulnerable to length extension: given $H(m)$, an attacker can compute $H(m \| \text{pad} \| m')$ for any $m'$ without knowing $m$. SHA-3 (Keccak, sponge construction) is immune to this.

What is a random oracle and why is the random oracle model used in cryptographic security proofs?

<showhide>
A **random oracle** is an idealized hash function that returns a uniformly random (but consistent) output for each new input. In the **random oracle model (ROM)**, security proofs treat the hash function as a truly random function, accessible only as a black box. The ROM allows proving security of many practical constructions (RSA-OAEP, HMAC, etc.) that lack proofs in the standard model. The caveat: the ROM is an idealization—no real hash function is a random oracle, and some schemes secure in the ROM are insecure when instantiated with specific hash functions.
</showhide>
