# Divide & Conquer: Van Emde Boas Trees

**Van Emde Boas (vEB) trees** are a data structure that supports predecessor, successor, insert, delete, and search on a universe $\{0, 1, \ldots, u-1\}$ of integers in $O(\log \log u)$ time per operation. This dramatically beats the $O(\log n)$ of balanced BSTs when $u$ is polynomial in $n$.

The key idea is a recursive decomposition of the universe. A vEB tree on universe size $u$ is split into $\sqrt{u}$ **clusters** of size $\sqrt{u}$ each, plus a **summary** structure of size $\sqrt{u}$ tracking which clusters are non-empty. Minimum and maximum elements are stored explicitly. The structure recurses: each cluster and the summary are themselves vEB trees on universe size $\sqrt{u}$.

Each operation recurses at most once into a cluster of size $\sqrt{u}$ and once into the summary of size $\sqrt{u}$, but the two recursive calls are not simultaneous—only one is made. This gives the recurrence $T(u) = T(\sqrt{u}) + O(1)$, which solves to $T(u) = O(\log \log u)$ via the substitution $m = \log u$, giving $S(m) = S(m/2) + O(1) = O(\log m)$.

## Structure of vEB Trees

A vEB tree $V$ on universe $[0, u)$ stores:
- $V.\min$ and $V.\max$: the minimum and maximum elements (not stored in any cluster).
- $V.\text{summary}$: a vEB tree on $[\,0, \sqrt{u}\,)$ tracking non-empty clusters.
- $V.\text{cluster}[i]$ for $i = 0, \ldots, \sqrt{u}-1$: vEB trees on $[\,0, \sqrt{u}\,)$.

Elements are mapped: integer $x$ goes to cluster $\lfloor x / \sqrt{u} \rfloor$ at offset $x \bmod \sqrt{u}$.

<checkyourself>
Why are min and max stored explicitly (not in any cluster)?

<showhide>
Storing min and max explicitly enables $O(1)$ access and allows insert/delete to avoid a recursive call in the "base" case where the structure has 0 or 1 elements. In particular, inserting into an empty vEB tree just sets min = max = x without recursion. This is what keeps the number of recursive calls per operation to 1 (not 2), making the $T(u) = T(\sqrt{u}) + O(1)$ recurrence valid.
</showhide>

## Predecessor and Successor Queries

To find the **successor** of $x$ in vEB tree $V$:
1. Compute $i = \lfloor x/\sqrt{u} \rfloor$, $j = x \bmod \sqrt{u}$.
2. If the max of cluster $i$ is $> j$, recurse into cluster $i$ for successor of $j$.
3. Otherwise, find the next non-empty cluster $i' > i$ using the summary, and return $\sqrt{u} \cdot i' + V.\text{cluster}[i'].\min$.

Only one recursive call of each type is made (cluster or summary), keeping the recurrence $T(u) = T(\sqrt{u}) + O(1)$.

What is the difference between vEB trees and $x$-fast tries / $y$-fast tries in terms of space and query time?

<showhide>
vEB trees use $O(u)$ space and $O(\log \log u)$ time per operation. $x$-fast tries use $O(n \log u)$ space and $O(\log \log u)$ per predecessor/successor. $y$-fast tries use $O(n)$ space and $O(\log \log u)$ expected time by grouping elements into balanced BSTs of size $\Theta(\log u)$ indexed by an $x$-fast trie on their minimums. $y$-fast tries are the most space-efficient while matching the query time.
