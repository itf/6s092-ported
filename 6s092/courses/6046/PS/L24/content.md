# Cache-Oblivious Algorithms: Searching & Sorting

This final lecture covers cache-oblivious algorithms for **searching** (static dictionaries) and **sorting**, completing the cache-oblivious algorithmic toolkit. The key data structure is the **van Emde Boas layout** for binary search trees, and the key algorithm is **cache-oblivious sorting** via funnel sort.

A **cache-oblivious BST** using the van Emde Boas layout achieves $O(\log_B N)$ cache misses per search—optimal. The layout recursively stores the top half of the tree followed by the bottom subtrees. For a tree of height $h = \log N$: the top $h/2$ levels are stored contiguously (size $2^{h/2} \approx \sqrt{N}$), then each subtree of size $\sqrt{N}$ is stored contiguously. This ensures that any root-to-leaf path hits $O(\log_B N)$ cache blocks.

**Funnelsort** achieves $O(N/B \cdot \log_{M/B}(N/B))$ cache misses—matching the optimal cache-aware merge sort. It uses **$k$-mergers** (lazy binary trees that merge $k$ sorted sequences) in a cache-oblivious way. The analysis relies on the "filling lemma": a $k$-merger outputs $k^2$ elements with $O(k^2/B + k\sqrt{k})$ cache misses.

## Van Emde Boas Layout for Cache-Oblivious BST

The **van Emde Boas layout** stores a complete binary tree of height $h$ as follows: if $h = 1$, store the root. Otherwise, let $h' = \lfloor h/2 \rfloor$; store the top subtree (height $h'$, $2^{h'} - 1$ nodes) recursively, then store each of the $2^{h'}$ bottom subtrees (each of height $h - h'$) recursively.

**Cache complexity for search**: a root-to-leaf path has length $\log N$. The path crosses $O(\log N / \log \sqrt{B})$ = $O(\log_B N)$ "recursive boundaries," each incurring one cache miss (when the path transitions from one recursively-stored chunk to another). Total: $O(\log_B N)$ misses per search.

<checkyourself>
Why does a sequential array layout (BFS order) for a static BST give $O(\log N)$ cache misses per search, while the van Emde Boas layout gives $O(\log_B N)$?

<showhide>
In BFS-order layout, the tree's $\log N$ levels are stored level by level. For large $N$, the bottom levels are far apart in memory; accessing a node at level $\ell$ may cause a cache miss if level $\ell-1$ is no longer in cache. In the worst case, each of the $\log N$ levels causes a miss, giving $O(\log N)$. In the van Emde Boas layout, subtrees of size $\Theta(B)$ are stored contiguously, so once a subtree's root is in cache, its entire depth-$\log B$ portion requires no further misses. Only $O(\log N / \log B) = O(\log_B N)$ such subtrees are traversed, giving $O(\log_B N)$ misses.
</showhide>

## Cache-Oblivious Sorting: Funnelsort

**Funnelsort** sorts $N$ elements using $O(N/B \cdot \log_{M/B}(N/B))$ cache misses—the same as cache-aware merge sort (optimal). Algorithm: recursively sort $N^{1/3}$ groups of $N^{2/3}$ elements each (each group fits in cache once small), then merge using a $\sqrt[3]{N}$-merger.

A **$k$-merger** lazily merges $k$ sorted sequences: it is a complete binary tree of buffers, where internal nodes merge their children's outputs. Cache-obliviousness follows from the recursive buffer structure, which matches the memory hierarchy automatically.

**Sorting lower bound**: any comparison-based sort in the cache model requires $\Omega(N/B \cdot \log_{M/B}(N/B))$ cache misses. This matches funnelsort—it is optimal.

<checkyourself>
Summarize the key advantages of cache-oblivious algorithms over cache-aware (blocked/tiled) algorithms.

<showhide>
**Cache-oblivious advantages**: (1) **Portability**: the same code works optimally on all machines and cache configurations without tuning $M$ or $B$. (2) **Multi-level optimality**: a cache-oblivious algorithm is automatically optimal at every level of a multi-level cache hierarchy (L1, L2, L3, RAM), whereas a cache-aware algorithm tuned for one level may be suboptimal at others. (3) **Simplicity of analysis**: the ideal cache model gives clean bounds. **Disadvantages**: cache-oblivious algorithms often have larger constants and more complex implementations than well-tuned cache-aware algorithms. In practice, carefully tuned BLAS libraries often outperform cache-oblivious implementations for specific hardware.
</showhide>
