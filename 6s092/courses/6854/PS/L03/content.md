# Consistent Hashing

Consistent hashing solves the cache assignment problem: how to map items to caches such that when a cache is added or removed, only a minimal number of items need to be reassigned.

## The Ring Construction

Map both items and caches to points on a circle $[0,1)$ uniformly at random. Each item is stored in the **nearest cache clockwise**. When a cache is added or removed, only items in the arc between the new/removed cache and its predecessor need to move.

<checkyourself>
With $n$ caches and $m$ items placed uniformly on the circle, how many items move on average when one cache is added?

<showhide>
A new cache is placed at a uniformly random position on the circle. It "claims" the arc between itself and its predecessor. The fraction of the circle it covers is $1/n$ in expectation (since $n$ caches divide the circle into arcs). Therefore, on average $m/n$ items need to move. This is optimal—we can't do better than moving the items that actually change ownership.
</showhide>
</checkyourself>

## Virtual Nodes

To improve load balance, each cache is given $O(\log n)$ **virtual nodes** (random positions on the circle). This ensures that with high probability, each cache's load fraction is within $O(\log n / n)$ of $1/n$.

<checkyourself>
Explain why consistent hashing provides a good solution for a web CDN (Content Delivery Network) that periodically adds or removes cache servers.

<showhide>
In a CDN, requests for a URL must go to the cache holding that URL's content. With consistent hashing: (1) each URL maps to exactly one cache deterministically given the current configuration; (2) when a cache server is added, only $O(m/n)$ URLs need to be fetched and stored on the new server; (3) when a server fails, only the URLs it was responsible for need to be re-fetched (from the origin or another server). This minimizes cache misses during topology changes, which is crucial for high-traffic CDNs.
</showhide>
